# 汇智云平台：现状到目标的实施路径

## 1. 文档目的

本文档不再讨论理想架构本身，而是讨论如何从当前系统状态逐步演进到目标架构。

目标不是一次性重构，而是以最小风险、最强兼容、最清晰边界的方式，把现有 `account / foundation / codocs / aims` 逐步收敛到新的平台内核。

本文档服务于以下问题：

- 当前哪些系统是既有事实，不能绕开
- 哪些能力应先冻结契约，再逐步替换实现
- 哪些模块适合先试点，哪些模块必须兼容优先
- 从“现有账号中心”走向“平台控制面”的实际路径是什么

---

## 2. 当前系统现状判断

### 2.1 `account`

当前 `account` 已经承担了平台管理、用户、角色、应用、资源等多项职责，但它的模型仍然偏“账号中心 + 权限后台”，尚未完全成为平台级 Control Plane。

其主要问题是：

- 历史角色语义混杂，岗位、职责、管理员角色交织在一起
- 资源权限体系已存在，但模板、覆盖、scope 等新模型尚未落地
- 现有接口更偏应用内管理后台使用，不是为“平台下发授权结果”设计
- 当前命名和职责仍然更像 `account`，而不是 `platform`

### 2.2 `foundation`

`foundation` 已经具备成为平台共享层的基础条件，尤其是：

- 已经在多个应用中承担公共 UI / 认证 / 权限消费逻辑
- 适合作为统一 SDK、统一路由守卫、统一服务端鉴权封装的宿主
- 是把平台协议真正落到各应用接入层的最佳位置

因此，未来平台对应用的通用能力，不应该散落在各 app 中，而应收口到 `foundation`。

### 2.3 `codocs`

`codocs` 已经上线，是现网兼容性最敏感的模块之一。

这意味着：

- 它不适合作为第一批大改造对象
- 它适合成为“兼容接入样板”，而不是“模型试验田”
- 任何影响登录、权限、菜单、文档访问链路的改造，都必须先通过兼容适配完成

### 2.4 `aims`

`aims` 目前处于更适合重构的阶段。

它的优势是：

- 业务域比较清晰
- 正在梳理权限模型
- 还没有形成无法动弹的历史包袱
- 天然需要“平台权限 + 项目上下文范围”两段式授权

因此，`aims` 最适合作为第一批完整接入新模型的业务应用。

## 3. 目标架构的落地原则

从现状走向目标，不应直接以“替换旧系统”为目标，而应遵循以下原则：

### 3.1 先冻结协议，再替换实现

优先冻结以下契约：

- 平台对应用提供的认证协议
- 平台对应用提供的授权结果结构
- 平台与 runtime 的 bundle / heartbeat / revocation 接口
- app manifest 结构
- foundation SDK 能力边界

协议先定，内部实现才能逐步切换，而不把现有系统拖进持续反复的重构。

### 3.2 先做平台能力，再做平台品牌切换

第一阶段不急于把 `account` 改名或整体迁到 `hzy_platform`。

更现实的路径是：

- 先在现有系统中增量引入平台能力
- 等核心协议和运行时模型稳定后，再决定库、服务、命名层面的重构

也就是说，先完成“平台化能力”，再完成“平台化命名与拆分”。

### 3.3 先做新模型消费端，再做旧模型清理

优先做的是：

- `foundation` 消费新协议
- `aims` 接入新模型
- `codocs` 获得兼容桥接

而不是先大规模删除旧表、旧接口、旧角色。

### 3.4 兼容优先顺序明确

第一批深度接入顺序建议固定为：

1. `account` / 平台后台
2. `foundation`
3. `aims`
4. `codocs`
5. 其他应用

---

## 4. 总体实施阶段

建议整体按六个阶段推进。

### Phase 0：冻结目标契约

目标：把未来平台的关键契约钉死，不再让现有系统继续发散。

输出物：

- 目标架构文档
- Identity Plane 设计
- Control Plane API 契约
- Foundation SDK 契约
- License / Capability 文档
- App Manifest 规范
- 现状到目标实施路径文档

本阶段要求：

- 新应用不得再直接对接 CAS 语义
- 新权限逻辑不得再自定义一套返回结构
- 新 app 的资源定义必须向 manifest 结构靠拢
- `foundation` 的新增公共能力必须按平台协议思路建设

### Phase 1：在现有 `account` 中增量引入平台数据模型

目标：先不拆库、不改名，在现有 `account` 基础上承接第一版平台控制面数据。

第一批建议新增/扩展的数据能力包括：

- 权限模板
- 模板绑定
- 模板覆盖
- 角色 scope
- 最小主体目录 `subjects`
- 部署模型 `deployments`
- 应用 manifest 元数据
- policy bundle 元数据
- revocation 元数据
- license / capability 基础表

这一阶段的关键原则：

- 优先增量建模
- 不主动触碰 `codocs` 现网依赖表结构
- 不急着做 `hzy_platform` 分库
- 不急着替换全部旧接口

### Phase 2：补齐最小可用的 Identity Plane

目标：先把“平台对下统一身份输出”做起来。

第一版 Identity Plane 应最小支持：

- 上游接入 `CAS / 企业微信 / GitLab OIDC / 通用 OIDC`
- 对下游应用统一输出平台 token
- 提供 JWKS
- 提供 refresh / logout
- 提供 revocation 拉取能力
- 提供 heartbeat 与 license 状态查询
- 提供 tenant 级 policy bundle 下载

这一阶段不追求：

- 浏览器侧全量 SDK
- 多语言 SDK
- 用户级增量 bundle
- 全量 session 中心化改造

### Phase 3：让 `foundation` 成为新模型统一接入层

目标：把平台协议真正落到各应用共用层。

这一阶段，`foundation` 需要新增或收口的能力包括：

- token 验签
- bundle 加载与缓存
- revocation 校验
- heartbeat 客户端
- 权限快照加载
- `checkPermission()`
- 菜单过滤
- 路由守卫
- 服务端鉴权封装
- app manifest 消费能力

这里的核心不是 UI，而是把 `foundation` 从“公共组件集合”推进为“平台接入 SDK + Layer”。

### Phase 4：以 `aims` 作为第一批完整接入应用

目标：把 `aims` 作为新授权模型和新接入模型的试点。

`aims` 第一批要完成的事情包括：

- 使用平台 token 与 foundation SDK
- 不再直接理解 CAS 语义
- 资源定义改成 manifest 驱动
- 应用级权限由平台提供
- 项目级数据范围仍由 `aims` 自己解释
- 服务端接口统一走平台权限 + 本地 scope resolver

`aims` 的目标不是“先完全做完所有功能”，而是“先完整跑通一条新模型链路”。

### Phase 5：为 `codocs` 做兼容迁移

目标：在不影响现网的前提下，让 `codocs` 逐步接上新平台。

建议顺序：

1. 先兼容接入平台 token
2. 再兼容 foundation 新鉴权封装
3. 再把资源定义整理为 manifest
4. 最后再逐步替换旧权限查询链路

这里必须坚持一个原则：

- `codocs` 先做协议兼容
- 不在第一阶段强推其内部权限模型重写

### Phase 6：平台后台与治理能力收口

目标：把平台真正做成可运营的 Control Plane。

这一阶段要补齐的是：

- 部署管理
- 应用接入管理
- License 管理
- Capability 管理
- 模板分配管理
- policy bundle 发布与版本管理
- heartbeat 监控
- revocation 运维
- 审计日志与授权追踪

这一步完成后，平台才真正从“账号后台”升级为“控制面”。

---

## 5. 推荐的优先级与依赖关系

为了降低风险，建议以下依赖顺序固定：

1. 先定协议文档
2. 再建平台增量数据模型
3. 再做 Identity Plane 最小实现
4. 再做 Foundation SDK
5. 再接 `aims`
6. 再做 `codocs` 兼容
7. 最后再考虑 `account` 是否演进为 `hzy_platform`

这意味着一个非常明确的判断：

- `hzy_platform` 不是第一阶段目标
- “平台能力成型”才是第一阶段目标

---

## 6. 当前最值得立即启动的实施范围

如果只做第一轮实际落地，建议范围收敛为以下四项：

### 6.1 在 `account` 中落第一版平台数据模型

只做增量表和增量接口，不做大迁移。

### 6.2 在 `foundation` 中落第一版平台 SDK 能力

先支持：

- token 验签
- bundle 拉取
- 权限判断
- 菜单 / 路由 / 服务端统一鉴权

### 6.3 在 `aims` 中做第一条完整接入链路

先打通：

- 登录后身份消费
- 应用权限判断
- 项目级 resolver
- 服务端权限收口

### 6.4 为 `codocs` 定兼容边界，不立刻强改

先产出兼容接入方案，不急于立即切换。

---

## 7. 关键风险

### 7.1 现有 `account` 继续发散

如果在新模型落地前，`account` 还持续新增新的旧式接口和权限写法，会导致迁移成本不断上升。

### 7.2 `foundation` 继续仅作为 UI 公共层使用

如果不把它及时推进为平台协议统一接入层，后续每个 app 都会继续各自实现一套权限接入。

### 7.3 过早动 `codocs`

`codocs` 的角色是兼容样板，不是第一批重构对象。过早强改，收益小，风险高。

### 7.4 过早讨论分库/改名/大迁移

在平台协议、Identity Plane、SDK、首个试点 app 都未跑通前，过早进入 `hzy_platform` 分库迁移，会把项目带偏。

---

## 8. 里程碑定义

建议按以下里程碑衡量推进，而不是按“是否改完旧系统”衡量。

### Milestone A：协议冻结

满足条件：

- OIDC-first 策略确定
- API 契约确定
- SDK 契约确定
- manifest 规范确定

### Milestone B：平台最小内核可运行

满足条件：

- `account` 已具备新数据模型
- Identity Plane 最小实现可用
- bundle / revocation / heartbeat 可跑通

### Milestone C：首个应用完整接入

满足条件：

- `foundation` 已消费新模型
- `aims` 跑通完整登录与授权链路
- `aims` 服务端权限已收口

### Milestone D：现网应用兼容接入

满足条件：

- `codocs` 可使用平台 token
- `codocs` 登录与权限不受影响
- 新旧接口可并存

### Milestone E：平台后台具备运营能力

满足条件：

- 部署、license、bundle、模板、审计可管理
- 平台真正具备控制面属性

## 10. 结论

从现状走向目标，不应以“大迁移”开局，而应以“冻结协议 + 增量建模 + foundation 收口 + aims 试点 + codocs 兼容”为主线。

这条路径有三个核心判断：

- 第一阶段先做平台能力，不先做平台品牌切换
- `aims` 是试点，`codocs` 是兼容对象
- `foundation` 是平台真正落到应用接入层的关键抓手

只要这三点不跑偏，现有系统就可以在不推倒重来的前提下，逐步收敛到目标平台架构。
