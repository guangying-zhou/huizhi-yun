# 汇智云平台内部应用鉴权机制分析与改进建议

> 基于 `huizhi-yun` 仓库当前实现的静态分析，重点评估平台内部应用之间的认证、授权、租户隔离、服务调用和 tenant-runtime 访问机制。

## 1. 执行摘要

汇智云当前的内部应用鉴权机制，不能简单概括为“太严格”。更准确的判断是：

> **安全边界设计总体合理，但工程实现过度复杂；授权配置重复，权限语义不统一，同时部分关键环节又不够精确。**

平台当前保留以下安全边界是必要的：

- 应用之间使用短期 Service JWT；
- 严格校验 `audience`；
- 绑定租户和部署上下文；
- 区分来源应用；
- 应用 BFF 与 tenant-runtime 分别鉴权；
- 写操作使用幂等键；
- 避免共享静态 API Key。

真正需要解决的问题不是减少鉴权层数，而是：

1. 同一授权关系被 Console、目标应用代码、SQL Seed 和设计文档重复维护；
2. 平台中并存多套 scope 命名方式，容易导致合法请求被拒绝；
3. `appCode` 同时表达来源应用和目标应用，存在语义冲突；
4. runtime 中的授权粒度普遍停留在应用级 `read/write`，过于宽泛；
5. 部分 Service API 代理可能使用目标应用自身的更高权限访问 runtime，形成权限放大；
6. actor、gateway 等关键上下文没有始终被可靠签名和校验。

因此，推荐方向是：

> **保留必要的双重信任边界，将授权关系集中到 Console，统一 scope/capability 语言，明确来源应用与目标应用语义，并以业务能力而不是应用级读写权限进行授权。**

---

## 2. 当前跨应用调用链

以 Altoc 调用 Finance 为例，一次内部调用大致会经过：

```text
用户
  ↓ 1. Altoc 用户权限校验 requirePermission
Altoc BFF
  ↓ 2. 向 Console 申请 Service Token
Console
  ↓ 3. 校验 service_client_grants
Finance BFF
  ↓ 4. 验证 JWT、audience、token_use
  ↓ 5. 部分接口再次检查 scope 和 allowedApps
  ↓ 6. Finance 获取 tenant-runtime Token
Finance tenant-runtime
  ↓ 7. 校验 audience、tenant、deployment、appCode、scope
业务命令
```

从架构上看，其中存在两个合理的信任边界：

1. **应用 A → 应用 B**；
2. **应用 B → 客户侧 tenant-runtime**。

因此，“双 Token”本身并不一定多余。应用运行时与客户数据运行时可能部署在不同网络和安全域中，分别鉴权是合理的。

问题在于，同一个授权关系在多处重复描述，而且各层对 scope、appCode、调用方和目标方的解释并不一致。

---

## 3. 当前机制中过度严苛且脆弱的部分

### 3.1 同一个调用许可被维护多次

以 Aims 读取 Assets 产品为例，授权关系可能同时存在于：

1. Console 的 `service_client_grants`；
2. Assets 目标接口中的 `allowedApps`；
3. SQL Seed 或迁移文件；
4. 设计文档和契约说明。

例如目标应用代码可能写成：

```ts
requireServiceScope(event, {
  scope: 'assets:read',
  allowedApps: ['aims']
})
```

这样一来，新增一个合法调用方 Altoc，通常需要同时：

- 修改 Console 授权；
- 增加或修改 SQL Seed；
- 修改 Assets 中的调用方白名单；
- 重新部署 Assets；
- 更新相关文档。

这会造成较高的维护成本，并增加配置不一致的概率。

更合理的模式是：

```text
目标接口只声明“需要什么能力”
Console 决定“哪些来源应用具备该能力”
```

目标接口只需要：

```ts
requireServiceCapability(
  event,
  'assets:product:read'
)
```

至于 Aims、Altoc 或 Finance 是否能获得该能力，应由 Console 的统一授权表决定。

`allowedApps` 可保留给极少数高危接口作为额外防线，但不应成为所有内部接口的标准配置。

---

### 3.2 Scope 表达方式不统一

仓库中存在多套权限表达方式，例如：

```text
assets:read
assets.read
tenant-runtime:assets:read
data-runtime:assets:read
```

Finance 相关代码中还可能出现：

```text
finance:read
finance.contracts.read
finance.contracts:read
tenant-runtime:finance:contracts.read
```

这不仅是命名风格问题，还会直接导致授权失败。

例如调用方请求：

```ts
scope: 'assets.read'
```

而 Console 的 scope 校验要求以：

```text
assets:
```

开头，则合法请求也会被判定为 `invalid_scope`。

建议统一采用一种格式：

```text
<目标应用>:<资源>:<动作>
```

例如：

```text
assets:delivery-package:read
finance:invoice-request:create
finance:contract-summary:read
aims:project-from-contract:create
altoc:receivable:mark-billable
```

不应再在应用间 API 中混用点号和冒号形式。

---

### 3.3 `appCode` 同时承担来源和目标两个含义

当前实现中，`appCode` 可能同时被解释为：

- 发起调用的来源应用；
- 当前请求要访问的目标 adapter。

例如 Finance 直接访问 Aims tenant-runtime：

```text
来源应用：finance
目标应用：aims
```

Console 签发 Token 时，`hzy.appCode` 可能写入来源应用 `finance`；但 tenant-runtime 又根据 `/v1/aims/**` 路径要求 `appCode=aims`。

如果 runtime 使用如下判断：

```text
Token appCode 必须等于目标 adapter
```

则即使 Console 明确授予 Finance 访问 Aims 的能力，请求仍可能因：

```text
finance != aims
```

而被拒绝。

这类语义冲突必须消除。

#### 推荐主路径

跨应用调用统一经过目标应用的 Service API：

```text
Finance → Aims Service API → Aims tenant-runtime
```

此时 Aims BFF 使用自身身份访问 Aims runtime，来源和目标语义一致。

#### 允许直连 runtime 时

Token 必须明确拆分：

```text
source_app = finance
target_app = aims
capability = aims:project-time-entry:read
```

runtime 应校验：

1. `target_app` 与请求路径一致；
2. `source_app` 具备相应 capability；
3. 不能再用单个 `appCode` 同时表达来源和目标。

---

### 3.4 双重 Token 合理，但由业务代码重复拼装

当前不同模块可能重复实现：

- 获取目标应用地址；
- 向 Console 申请 Service Token；
- 转发租户和部署上下文；
- 转发 gateway token；
- 获取 runtime URL 和 runtime token；
- 转发幂等键；
- 处理响应 envelope；
- 将下游错误映射为 502 或其他错误。

这会导致各应用实现略有差异，容易漏传 Header、错误处理不一致或出现安全绕过。

建议在 Foundation 中提供统一客户端：

```ts
callServiceApi(event, {
  targetApp: 'finance',
  capability: 'finance:invoice-request:create',
  path: '/api/v1/finance/invoice-requests',
  method: 'POST',
  body,
  idempotencyKey
})
```

由 Foundation 统一处理：

- Service Token 获取和缓存；
- 租户和部署上下文；
- 请求追踪；
- actor 委托；
- 幂等键；
- 错误归一化；
- 超时和重试策略。

---

## 4. 看似严格但实际上不够精确的部分

### 4.1 `scope_json.endpoints` 没有真正参与授权

Console 的授权记录可能包含：

```json
{
  "endpoints": [
    "/api/v1/finance/invoice-requests",
    "/api/v1/finance/invoice-requests/{code}/submit"
  ]
}
```

但实际签发 Token 时，只读取类似：

```text
resource_code
action
```

然后生成：

```text
finance:write
```

如果 endpoint 限制没有进入 Token，也没有在目标应用执行，则数据库配置看起来只授权了两个接口，实际却获得了整个 Finance 写权限。

这会产生“配置过程严格、最终权限宽泛”的反差。

建议二选一：

1. 真正执行 endpoint 和 method 条件；
2. 删除容易误导的 endpoint 配置，改为真实细粒度 capability。

推荐第二种，以业务能力为授权单位。

---

### 4.2 Runtime 权限普遍停留在应用级 `read/write`

除少数模块外，tenant-runtime 对通用路径的授权通常类似：

```text
GET    → app.read
其他   → app.write
```

例如拥有：

```text
altoc.write
```

理论上就可能拥有 Altoc 整个 runtime 的写能力。

这与上层复杂的授权配置并不匹配。

对于高价值业务命令，应拆分为明确 capability，例如：

```text
altoc:lead:convert
altoc:opportunity:transition
altoc:receivable:mark-billable
altoc:contract:finance-summary:sync
altoc:service_ticket:delivery-result:sync
```

应用自己的 BFF 可以被授予多个能力，但跨应用调用方只获得完成特定业务操作所需的最小能力。

---

### 4.3 Service API 代理可能形成权限放大

部分 `/service/**` 接口可能直接代理到 tenant-runtime。

如果入口只验证：

- Token 有效；
- audience 正确；

但没有明确校验该调用方是否拥有特定业务 capability，目标应用随后又使用自身更强的 runtime Token 访问数据，则可能形成：

```text
调用方使用较宽泛的应用 Token进入
→ 目标 BFF 使用自身更高权限访问 runtime
→ 调用方间接获得超出授权范围的数据或写能力
```

因此，目标应用在使用自己的 runtime 凭证前，必须先校验入站 capability。

---

### 4.4 tenant-runtime 应强制校验 `token_use=service`

内部服务调用 Token 应明确包含：

```text
token_use=service
```

tenant-runtime 不应只校验：

- JWT 签名；
- audience；
- tenant；
- deployment；
- appCode；
- scope。

还应明确拒绝用户 Token 或其他用途的 Token。

生产环境建议将以下 claim 设为必填：

```text
token_use
source_app
target_app
tenant_id
deployment_id
aud
capabilities
```

不能继续采用“claim 有值才比较、为空则放行”的宽松行为。

---

### 4.5 Actor 身份通过普通 Header 传递

当前用户身份可能通过：

```text
x-hzy-actor-uid
```

在内部应用之间传递。

这意味着目标应用必须完全信任来源应用不会伪造用户身份。

在内部可信应用中，这种方式可以作为过渡方案；但对于合同、财务和审批等高价值操作，更建议采用签名委托声明，例如：

```json
{
  "sub": "client:altoc.runtime",
  "act": {
    "sub": "user:zhou"
  }
}
```

或者使用 Console 签发的短期 on-behalf-of Token。

这样，以下信息都处于同一个签名 Token 中：

- 来源应用；
- 目标应用；
- 用户 actor；
- 授权能力；
- 租户和部署；
- 有效期。

---

### 4.6 Gateway 信任检查不一致

部分模块会同时校验：

```text
x-hzy-gateway = tenant-gateway
x-hzy-gateway-token = 预期内部 Token
```

但另一些 Foundation runtime client 可能只检查：

```text
x-hzy-gateway === tenant-gateway
```

如果应用原始地址可以被直接访问，单纯依赖一个字符串 Header 容易被伪造。

建议所有应用统一复用相同的 Gateway 校验方法：

1. 校验 gateway 标识；
2. 校验 gateway token 或 mTLS 身份；
3. 校验请求时间和防重放信息；
4. 不接受来自公网的伪造 runtime URL、runtime Token 或租户上下文。

---

## 5. 推荐的目标授权模型

建议将应用间授权关系统一收敛到 Console，建立单一事实源：

```text
caller_app
target_app
capability
tenant_scope
delegation_allowed
status
```

示例：

| 调用方 | 目标应用 | Capability |
|---|---|---|
| Altoc | Aims | `aims:project-from-contract:create` |
| Altoc | Finance | `finance:invoice-request:create` |
| Altoc | Finance | `finance:contract-summary:read` |
| Aims | Altoc | `altoc:receivable:mark-billable` |
| Finance | Altoc | `altoc:contract:finance-summary:sync` |
| Altoc | Assets | `assets:delivery-package:read` |

目标接口只声明：

```ts
requireServiceCapability(
  event,
  'assets:delivery-package:read'
)
```

统一帮助函数负责校验：

- JWT 签名；
- issuer；
- audience；
- `token_use=service`；
- 来源应用；
- 目标应用；
- tenant；
- deployment；
- capability；
- 委托 actor；
- Token 有效期和撤销状态。

不再由每个业务模块分别维护调用方白名单。

---

## 6. 推荐的 Token Claims

建议内部 Service Token 至少包含：

```json
{
  "iss": "https://console.example.com",
  "aud": "finance",
  "sub": "client:altoc.runtime",
  "token_use": "service",
  "source_app": "altoc",
  "target_app": "finance",
  "tenant_id": "tenant-001",
  "deployment_id": "deployment-001",
  "capabilities": [
    "finance:invoice-request:create"
  ],
  "act": {
    "sub": "user:zhou"
  },
  "jti": "...",
  "iat": 0,
  "exp": 0
}
```

其中：

- `source_app`：发起调用的应用；
- `target_app`：目标应用或目标 runtime adapter；
- `capabilities`：具体业务能力；
- `act`：代表哪位用户执行；
- `tenant_id`、`deployment_id`：租户和部署隔离；
- `token_use`：明确 Token 只能用于服务调用；
- `jti`：用于审计、撤销和防重放。

---

## 7. 哪些安全检查应继续保留

| 检查 | 建议 |
|---|---|
| Console 签发短期 Service JWT | 保留 |
| JWKS 签名验证 | 保留 |
| 严格 audience | 保留 |
| `token_use=service` | 保留并加强 |
| tenant / deployment 绑定 | 保留并改为必填 |
| 来源应用身份 | 保留 |
| 目标应用身份 | 明确拆分并保留 |
| 细粒度 capability | 加强 |
| 写操作幂等键 | 保留 |
| App BFF 与 tenant-runtime 分离鉴权 | 保留 |
| Gateway 身份验证 | 保留并统一 |
| 共享静态 API Key | 不应恢复 |
| 内网默认可信 | 不建议采用 |

简化鉴权不应以放松这些安全边界为代价。

---

## 8. 建议的统一基础设施接口

### 8.1 目标应用入口

```ts
const identity = await requireServiceCapability(
  event,
  'finance:invoice-request:create'
)
```

返回统一身份信息：

```ts
interface ServiceIdentity {
  sourceApp: string
  targetApp: string
  tenantId: string
  deploymentId: string
  capabilities: string[]
  actorUid?: string
  tokenId: string
}
```

### 8.2 调用方客户端

```ts
await callServiceApi(event, {
  targetApp: 'finance',
  capability: 'finance:invoice-request:create',
  path: '/api/v1/finance/invoice-requests',
  method: 'POST',
  body,
  idempotencyKey
})
```

### 8.3 tenant-runtime 调用

业务模块不应手工拼装 runtime Token 和 Header，而应使用：

```ts
await callTenantRuntime(event, {
  appCode: 'finance',
  capability: 'finance:invoice-request:create',
  path: '/v1/finance/invoice-requests',
  method: 'POST',
  body
})
```

由 Foundation 统一处理 Token、租户上下文和错误映射。

## 9. 改造优先级

### P0：修复逻辑冲突和安全缺口

1. 统一 scope/capability 格式，禁止继续混用 `assets.read`、`assets:read` 等形式；
2. 明确是否允许应用直接访问其他应用的 tenant-runtime；
3. 将 `source_app` 与 `target_app` 拆分，修复 `appCode` 语义冲突；
4. tenant-runtime 强制校验 `token_use=service`；
5. tenant、deployment、source_app、target_app 在生产环境必须存在；
6. 所有 `/service/**` 入口在代理 runtime 前校验具体 capability；
7. Foundation 统一校验真实 Gateway Token，而不只检查标识 Header；
8. 生产环境禁止默认 `auth=disabled`；
9. 静态 Token 仅作为明确配置的离线兼容方案。

### P1：删除重复配置和手写逻辑

1. Console grant 成为内部授权的唯一事实源；
2. 移除或自动生成业务代码中的 `allowedApps`；
3. 不再手工维护大量独立 SQL Seed 授权脚本；
4. 将授权关系纳入应用安装、升级和卸载生命周期；
5. 增加统一的 `callServiceApi()`；
6. 增加统一的 `requireServiceCapability()`；
7. 统一下游错误、审计和 tracing 格式。

### P2：升级为业务能力授权

1. 将 `altoc.write`、`finance.write` 等宽权限拆分为领域命令 capability；
2. 引入签名 actor 委托或 Token Exchange；
3. 提供授权关系管理和可视化页面；
4. 提供调用链诊断工具；
5. 建立跨应用契约测试；
6. 增加 Token 撤销、调用审计和异常告警。

---

## 10. 建议的契约测试清单

每条跨应用调用至少覆盖：

- 正确来源应用、正确 capability，调用成功；
- 缺少 capability，返回 403；
- audience 错误，返回 401；
- `token_use` 不是 service，返回 401；
- 来源应用错误，返回 403；
- 目标应用与路径不一致，返回 403；
- tenant 不一致，返回 403；
- deployment 不一致，返回 403；
- Token 过期，返回 401；
- actor 委托缺失或伪造，返回 401/403；
- Gateway Token 错误，拒绝请求；
- 幂等键重复，返回原结果而不是重复创建；
- 调用方被停用或授权撤销，立即拒绝；
- 下游 runtime 失败时，不泄露内部地址和敏感信息。

---

## 11. 预期收益

### 安全性

- 每个调用只获得最小业务能力；
- 来源应用、目标应用、用户 actor 和租户上下文均可追溯；
- 避免 BFF 代理造成权限放大；
- 减少 Header 伪造和 Token 用途混淆。

### 可维护性

- 一个授权关系只在 Console 维护一次；
- 新增调用方不需要修改并重新部署目标业务应用；
- scope 命名统一，减少合法请求被错误拒绝；
- 通用客户端减少重复代码。

### 可运营性

- 可以直接查看“谁能调用谁、具有什么能力”；
- 可以基于 capability 做审计和告警；
- 可以快速定位 401、403、租户不匹配和 audience 错误；
- 应用安装和卸载时可以自动建立或回收授权。

---

## 12. 最终结论

汇智云当前内部应用鉴权的安全目标并不过度，真正的问题是：

```text
认证边界合理
+ 授权关系重复维护
+ scope 语言不统一
+ 来源和目标语义冲突
+ runtime 权限粒度过宽
+ 部分关键上下文没有可靠签名绑定
```

因此，不建议简单减少 Token、取消 audience 或放宽租户校验，也不建议退回“内部网络默认可信”或共享 API Key。

推荐的最终方向是：

> **保留应用间和 tenant-runtime 两条必要信任边界，将授权集中到 Console，以明确的业务 capability 取代应用级 read/write，并由 Foundation 统一完成 Token、租户、actor、Gateway、幂等和错误处理。**

这会让平台从“门禁很多但规则容易冲突”，转变为：

> **边界清晰、授权最小、配置集中、调用可追踪。**
