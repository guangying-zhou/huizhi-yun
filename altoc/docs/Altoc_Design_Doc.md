## Problem Statement

汇智云团队的公司是一家面向 ToB/ToG 的项目型组织（IT 服务商/系统集成商类型）。现有一套 OA 系统，能做客户、合同、回款的基础台账录入，但存在以下核心问题：

1. **台账 ≠ 管理**：数据只是记录下来了，但没有工作流驱动、没有阶段推进、没有自动联动
2. **链路断裂**：客户→商机→报价→合同→回款之间没有打通，管理层看不到全景
3. **管理层决策盲区**：老板/经营管理层是最痛的人——看不到真实的销售漏斗、回款预测、经营健康度，做决策靠"问人"而不是"看系统"
4. **销售团队缺乏提效工具**：销售人员在多个系统间切换，没有工具帮助他们更高效地推进商机

Altoc 的目标是成为面向项目型销售的 AI 辅助 LTC（Lead to Cash）经营平台，从线索到回款形成完整闭环。

## Demand Evidence

- **内部需求验证**：团队自身就是目标用户，每天都在经历这些痛点
- **现有系统不足**：OA 台账只能记录，不能驱动流程和产生洞察
- **管理层是最痛的角色**：老板和经营管理层无法从现有系统获得决策所需的可信数据
- **行业普遍痛点**：项目型 ToB/ToG 销售是一个大量中小企业共同面临的场景，现有 CRM 工具多面向标准化销售，不适配项目型销售的长周期、多角色、重审批特点

## Status Quo

- 现有一套 OA 系统，覆盖客户、合同、回款的基础台账
- 台账层面的功能：能录入，但停留在"记录"层面
- **缺失的部分**：
  - 销售阶段管理与工作流
  - 商机漏斗与赢率预测
  - 报价审批与毛利管控
  - 合同→交付→回款的自动联动
  - 经营看板与销售预测
  - AI 辅助（智能提醒、纪要结构化、风险识别）

## Target User & Narrowest Wedge

### Primary User: 老板/经营管理层

- **核心诉求**：看见真实的销售漏斗、回款预测、经营健康度
- **痛点**：做决策靠"问人"而不是"看系统"，数据不可信、不及时、不完整
- **成功标准**：打开 Altoc 就能看到"今天该关注什么"——哪些商机有风险、哪些合同快到期、回款节奏是否正常

### Secondary User: 销售团队

- **核心诉求**：提高销售效率，减少重复劳动
- **痛点**：多系统切换、手工录入、不知道下一步该做什么
- **成功标准**：在一个系统里完成商机推进、报价、合同提交，系统告诉他"下一步该做什么"

### Narrowest Wedge

客户→线索→商机→报价→合同→回款的主链路 MVP。线索是商机的入口，必须包含在 MVP 中以确保数据来源可追溯。不做投标管理、不做交付里程碑联动、不做进销存。先把"从线索到收到钱"这条线跑通。

## Constraints

1. **技术栈**：Nuxt 4 + Nuxt UI v4 + MySQL + Nitro API（与 monorepo 其他模块一致）
2. **身份认证**：复用 Account 模块（CAS SSO + 企业微信 OAuth）
3. **端口**：3003（已在 package.json scripts 中配置 `--port 3003`）
4. **已有基础**：认证中间件、权限体系、Account API 集成已就绪（altoc 目录已创建，基础代码已存在）
5. **已有文档**：详细的 PRD（README.md）、数据模型（Altoc_Data_Model.md）、MVP 功能清单（Altoc_MVP_Features.md）
6. **前端规范**：严格使用 Nuxt UI v4 组件和 Tailwind CSS，颜色必须用 primary/success/warning/error 等语义名

## Premises

1. **双核心价值**：Altoc 同时服务管理层（看见真相）和销售团队（提效），两者同等重要。MVP 不能只做看板而忽略销售工作流。
2. **独立平台**：Altoc 是独立经营平台，不依赖现有 OA。通过 Account 模块共享身份和组织数据。
3. **数据录入成本必须极低**：AI 辅助录入是关键差异化。如果销售不录数据，管理层就什么都看不到。
4. **先有数据，再有 AI**：P0 先打通主链路产生结构化数据，P1 再叠加 AI 能力。
5. **遵循已有设计**：PRD、数据模型、功能清单已经过充分设计，MVP 实现应基于这些文档。

## Approaches Considered

### Approach A: 商机驱动型（Minimal Viable）

- **Summary**: 只做商机管理 + 经营看板
- **Effort**: M (2-3 周 CC 开发)
- **Risk**: Low
- **Pros**: 最快出版本；聚焦核心
- **Cons**: 链路不完整（Completeness 5/10）；管理层只能看到漏斗，看不到合同和回款
- **Verdict**: 走捷径省不了多少时间，但失去了"全链路"这个核心卖点

### Approach B: 主链路贯通型（Ideal Architecture）✅ CHOSEN

- **Summary**: 客户→商机→报价→合同→回款全链路 + 经营看板
- **Effort**: L (4-6 周 CC 开发[注: CC = Claude Code 辅助单人开发] / 人工团队约 3-4 个月)
- **Risk**: Med
- **Pros**: 完整链路，真正可用；管理层看到全景；一次性做完不补打补；有数据基础支撑后续 AI
- **Cons**: 周期稍长；数据模型较复杂
- **Completeness**: 9/10
- **Reuses**: 已有 auth + Account 集成 + 已设计的数据模型、PRD、功能清单

### Approach C: AI-First 型（Creative/Lateral）

- **Summary**: AI 销售助手先行，再补业务流程
- **Effort**: M-L
- **Risk**: High
- **Pros**: 差异化明显
- **Cons**: AI 依赖结构化数据，没数据就没 AI（Completeness 6/10）
- **Verdict**: 本末倒置。先有数据，再有智能。

## Recommended Approach

**选择 Approach B: 主链路贯通型**

### 实现范围

基于已有 PRD 和功能清单，MVP 一期 P0 范围：

#### 核心模块

| 模块 | 关键功能 | 服务对象 |
|------|---------|---------|
| 客户管理 | 企业/政府客户、联系人、标签分类、跟进记录 | 销售 |
| 线索管理 | 线索录入、分配、来源归因、转商机 | 销售 |
| 商机管理 | 阶段推进、漏斗视图、赢率、跟进记录 | 销售+管理层 |
| 报价管理 | 报价创建、审批、毛利计算、版本追踪 | 销售+管理层 |
| 合同管理 | 合同创建、审批、付款条款、变更管理 | 销售+管理层 |
| 回款管理 | 回款计划、开票申请、到账登记、逾期预警 | 财务+管理层 |
| 经营看板 | 漏斗分析、预测、回款趋势、应收预警 | 管理层 |
| 系统设置 | 销售阶段、审批规则、权限配置 | 管理员 |

#### 信息架构

一级导航：首页 / 客户 / 线索 / 商机 / 报价 / 合同 / 回款 / 看板 / 设置

#### 核心数据流

```
线索 ──转化──> 商机 ──报价──> 报价单 ──签约──> 合同 ──回款──> 回款记录
  │              │              │             │            │
  └── 客户 ──────┴──────────────┴─────────────┴────────────┘
                                                           │
                                              经营看板 <────┘
```

### 状态流转设计

每个核心实体的生命周期状态机：

| 实体 | 状态流转 | 允许回退 |
|------|---------|---------|
| 线索 | 新建 → 跟进中 → 已转商机 / 已关闭 | 已关闭 → 新建（重新激活） |
| 商机 | 初步接触 → 需求确认 → 方案报价 → 商务谈判 → 赢单 / 输单 | 任意前向阶段可回退；赢单/输单 → 重新打开 |
| 报价 | 草稿 → 待审批 → 已批准 / 已驳回 → 已发送 → 已接受 / 已拒绝 | 已驳回 → 草稿（修改重提）；已拒绝 → 草稿 |
| 合同 | 草稿 → 待审批 → 已批准 / 已驳回 → 已签署 → 执行中 → 已完成 / 已终止 | 已驳回 → 草稿；已终止需要特殊审批 |
| 回款 | 待开票 → 已开票 → 待回款 → 已到账 / 逾期 | 逾期 → 已到账（收到款后）；争议状态需人工处理 |

所有状态变更记录到 `status_change_log` 表，包含：实体类型、实体ID、旧状态、新状态、操作人、操作时间、备注。

### 审批机制（MVP 方案）

MVP 采用**简单状态机 + 角色审批**，不做通用审批流引擎（通用引擎留作 P1）：

- 报价审批：销售提交 → 销售经理审批（折扣超阈值时需更高级别审批）
- 合同审批：销售提交 → 销售经理审批 → 管理层审批（金额超阈值时）
- MVP 硬编码以下审批阈值（不可在 UI 配置，P1 再做可配置）：
  - 报价折扣 > 15% → 需销售经理审批
  - 报价折扣 > 30% → 需管理层审批
  - 合同金额 > 50万 → 需管理层审批
  - 合同金额 ≤ 50万 → 销售经理审批即可

### 附件管理

所有业务实体支持附件上传，统一通过 Aliyun OSS 存储：

- 存储路径：`altoc/{entity_type}/{entity_id}/{filename}`
- 文件大小限制：单文件 50MB
- 允许类型：文档（pdf/doc/docx/xls/xlsx）、图片（jpg/png）、压缩包（zip/rar）
- 通用 `document_link` 表保存 Codocs 文档引用；合同文本和扫描件通过 `contract_text` / `legacy_contract_scan` 等 `link_type` 关联，不再暴露本地 `/api/v1/attachments` 上传主路径

### 审计日志

所有核心业务操作记录审计日志：

- `audit_logs` 表：id, entity_type, entity_id, action(create/update/delete/status_change), old_value(JSON), new_value(JSON), operator_id, operator_name, ip_address, created_at
- MVP 阶段记录：创建、修改、删除、状态变更
- 不做实时查询界面（P1），但数据必须记录

### 通知机制（MVP 方案）

MVP 采用调用account提供的API发送**企业微信通知**，不做邮件推送（留作 P1）：

- 触发场景：商机超过 N 天未跟进、报价/合同待审批、回款逾期、合同即将到期
- 首页待办区展示未读通知

### 组织与团队模型

P1 引入销售团队体系，作为数据权限、业绩统计和审批流的组织基础：

**团队类型：**
- `sales` 销售团队 — 关联客户、商机，承担业绩目标
- `business` 商务团队 — 配合招投标、合同谈判
- `presales` 售前支持团队 — 方案支持、技术演示

**团队角色：**
- `senior_manager` 高级销售经理 — 看团队及下级团队全部数据，审批重大合同
- `manager` 销售经理 — 看本团队全部数据，审批报价和一般合同
- `assistant` 助理 — 本团队数据只读为主，可录入不可审批
- `member` 成员 — 仅看本人数据

**数据权限推导：** 业务对象通过 `owner_user_id` → `sales_team_member` → `sales_team` 关联。经理角色看团队数据，成员只看自己的。

**P1 实现范围：** 团队 CRUD + 成员管理 + 按团队筛选负责人。
**后续扩展：** 销售梯队等级、多人协作、团队目标、跨团队审批。

详细数据模型见 `Altoc_Data_Model.md` 第 12A 节。

### 架构决策：单租户

MVP 采用单租户架构（自用优先）。多租户留作 P2 设计：
- MVP 不添加 tenant_id 字段，避免过早引入复杂度
- P2 阶段统一设计多租户方案（可能是 tenant_id 列、schema 隔离或其他方案），届时通过数据库迁移添加
- 文档记录多租户意图，确保 MVP 的表结构设计不引入反多租户的硬依赖

### 性能考量

MVP 假设数据规模：每个实体 <1000 条记录。在此规模下：
- 看板聚合查询使用实时 SQL 查询（无需预计算）
- 关键查询字段建立索引：status, owner_id, customer_id, created_at, due_date
- 列表页默认分页 20 条/页

### 经营看板 KPI 定义

| 指标 | 计算公式 | 说明 |
|------|---------|------|
| 商机赢单率 | 赢单数 / (赢单数+输单数) × 100% | 不含进行中商机 |
| 平均销售周期 | Σ(赢单日期-创建日期) / 赢单数 | 单位：天 |
| 漏斗金额 | Σ(进行中商机预计金额) | 按阶段分组展示 |
| 加权预测 | Σ(商机金额 × 阶段赢率) | 阶段赢率可配置 |
| 回款达成率 | 本期实际回款 / 本期计划回款 × 100% | 按月统计 |
| 逾期应收 | Σ(已逾期未回款金额) | 按逾期天数分级 |
| DSO | 应收账款余额 / (年度收入/365) | 日销售未收款天数 |
| 报价转化率 | 已接受报价数 / 总报价数 × 100% | 排除草稿状态 |

- **销售首页**: 待办列表（按优先级排序）→ 今日跟进任务 → 我的商机概览 → 待审批项

- 每列：阶段名 + 商机数 + 金额合计
- 卡片：客户名 + 商机金额 + 赢率 + 负责人头像
- 拖拽到新阶段 → 乐观更新 + 成功 Toast

#### 看板页面

- KPI 卡片行（4 列）→ 漏斗柱状图 → 回款折线图 → 风险列表
- 不用饼图。异常指标用 error 色高亮。
- 无数据时 KPI 显示 "--" 而非 0

#### 颜色语义

- `primary`: 主操作、品牌色
- `success`: 赢单、已到账、已签署
- `warning`: 待审批、即将到期
- `error`: 逾期、输单、异常指标
- `info`: 信息提示
- `gray`: 次要文本、禁用状态

## Resolved Decisions

| 问题 | 决策 | 理由 |
|------|------|------|
| 审批流引擎 | MVP 用简单状态机+角色审批，P1 做通用引擎 | 降低 MVP 复杂度 |
| 多租户 | MVP 单租户，预留 tenant_id 字段 | 先跑通自用场景 |
| 通知机制 | MVP 站内通知，P1 做企业微信/邮件 | 最小化外部依赖 |
| 系统设置 | MVP 硬编码合理默认值，P1 做可配置 | 减少管理界面开发量 |

## Open Questions

1. **数据迁移**：现有 OA 中的客户/合同/回款数据是否需要迁移到 Altoc？如需迁移，需要了解 OA 的数据库结构。
2. **移动端**：MVP 是否需要考虑移动端适配（企业微信内嵌 H5）？建议 MVP 仅做 PC 端，响应式布局留作 P1。
3. **产品目录**：报价模块需要产品/服务目录作为基础数据，MVP 是预置还是需要管理界面？**注意：此问题需在 P0b 开始前（第2周末）决定，否则会阻塞报价模块开发。** -- 回答：产品目录对接assets（资产管理）模块，MVP阶段不需要管理界面，预置即可。

## Success Criteria

### 管理层视角

- [ ] 打开看板能看到实时的销售漏斗和商机分布
- [ ] 能看到未来 3 个月的回款预测
- [ ] 能看到逾期应收和风险客户
- [ ] 能看到各销售人员的商机推进质量

- [ ] 在一个系统内完成商机创建→报价→合同提交的全流程
- [ ] 每个商机都有明确的阶段和下一步动作
- [ ] 报价审批流在系统内完成，不需要线下沟通
- [ ] 合同签约后自动生成回款计划

### 系统层面

- [ ] 主链路（客户→线索→商机→报价→合同→回款）数据完整贯通
- [ ] 权限体系完整（角色+数据权限）
- [ ] 与 Account 模块的身份/组织数据正确同步
