## Problem Statement

Aims 前端 MVP 功能已基本实现（项目管理、工作项、缺陷、看板），但代码组织混乱导致后续开发效率低下。核心问题：重复常量散布在 5+ 个页面、只有 6 个组件（所有业务逻辑塞在页面里）、PIVR 三层架构在前端缺乏统一表达、表单验证缺失、布局文件过重、菜单结构与实际页面不匹配。

## Demand Evidence

- 开发团队和管理层都在推动 AIMS 尽快上线替代禅道
- 当前每新增一个页面就要重新定义状态映射、颜色映射、图标映射，开发效率低
- PIVR 模型（Preparation→Implementation→Validation→Refinement）是产品核心差异化，但前端没有清晰的组件对应
- Beta 阶段需要增加甘特图、报表、需求管理等大量页面，当前架构无法高效支撑

## Key Decision 1: 废弃迭代（Sprint）

**决策：** PIVR 里程碑作为工作项的唯一一级容器，迭代完全废弃。

**理由：**
- PIVR 模型已包含迭代的核心功能（P→I→V→R = 计划→执行→验证→回顾）
- 减少一个概念层级：从"里程碑→迭代→工作项"简化为"里程碑→工作项"
- PIVR 适配所有项目类型（瀑布、维保、售前），迭代只适用于敏捷
- 滚动计划模式的里程碑可替代 Sprint 的时间盒功能

**影响范围：**
- 删除旧迭代页面、旧迭代 store、旧迭代组件目录
- `plan.vue` 只管理里程碑（PIVR），不再包含迭代规划
- 看板按里程碑阶段组织，非按迭代组织
- 侧边栏菜单去掉"迭代"菜单项

## Key Decision 2: 精简工作项类型为三种

**决策：** 工作项类型从 5 种（epic/story/task/sub_task/bug）精简为 3 种：**需求(requirement)、任务(task)、缺陷(bug)**。

**理由：**
- Epic 是"跨迭代大特性"——废弃迭代后，里程碑本身就是容器，大需求通过 parent_id 嵌套表达层级
- Story 是敏捷专属术语，与 PIVR 模型脱钩，合并为"需求"
- Sub-task 不需要独立类型——数据库已有 `parent_id` 字段，任何任务设了 parent_id 就是子任务

**精简后的三种类型与 PIVR 三层架构对应：**

| 工作项类型 | 架构层 | 职责 |
|-----------|--------|------|
| 需求 (requirement) | 价值层 | 做什么——需求描述、验收标准、关联文档 |
| 任务 (task) | 执行层 | 怎么做——开发/测试/设计工作（parent_id 嵌套实现子任务） |
| 缺陷 (bug) | 质量层 | 哪里坏了——严重程度、复现步骤、修复验证 |

**影响范围：**
- `typeConfig` 从 5 种改为 3 种
- `WorkItemCard.vue` 差异化策略从 5 种改为 3 种
- 数据库 `work_items.type` 枚举调整为 `requirement/task/bug`
- 所有筛选器、状态选项对应调整

## Constraints

1. **不能中断现有功能** — 重构过程中所有已实现功能必须继续可用
2. **布局样式绝对不变** — Step 5 拆分布局时，视觉效果必须与重构前完全一致（像素级）
3. **技术栈锁定** — Nuxt 4 + Vue 3 + Nuxt UI V4 + Pinia + Tailwind CSS
4. **必须符合 PIVR 模型** — 里程碑固定 PIVR 四阶段，无迭代
5. **时间约束** — MVP 目标 2026 年 4 月中旬，重构不能显著推迟交付

## Premises

1. 核心问题不是功能缺失，而是架构混乱——"胖页面"模式是最大债务
2. PIVR 二层架构（里程碑→工作项）需要在前端有清晰的组件和数据流对应
3. 重构必须在继续开发新功能之前完成，否则技术债会继续累积
4. 目标是让新页面开发效率翻倍——开发者不需要再从头写状态映射和列表逻辑
5. 菜单组织必须科学合理，与 PIVR 工作流自然对应

## Approaches Considered

### Approach A: 常量+组件提取（Quick Win）
只做最小重构——提取重复常量和 3-5 个常用 UI 片段为组件。不改页面结构。
- Effort: S | Risk: Low | Completeness: 4/10

### Approach B: 前端架构标准化（CHOSEN）
系统性重构：共享常量、业务组件库、可复用 composables、Zod 验证、布局瘦身、菜单重组织。
- Effort: M | Risk: Med | Completeness: 8/10

### Approach C: 全面组件化+设计系统
在 B 基础上加 Storybook、完整类型系统、E2E 测试。
- Effort: L | Risk: Med-High | Completeness: 10/10

## Recommended Approach

### 方案 B：前端架构标准化

分 7 个步骤执行，每步独立可验证：

### Step 0: 清理迭代相关代码（~15 min CC）

**删除/清理：**
- 删除项目下的旧迭代页面

### Step 1: 共享常量与枚举配置（~30 min CC）

创建 `app/config/` 目录，集中管理所有业务常量：

```
app/config/
├── work-item.ts      # WorkItem 相关的所有映射
├── project.ts        # 项目状态、类型映射、模块开关
├── milestone.ts      # PIVR 阶段定义、颜色、图标
├── navigation.ts     # 菜单配置（全局视图 + 项目视图）
└── index.ts          # 统一导出
```

**work-item.ts 核心内容：**
- `statusConfig`: Record<WorkItemStatus, { label: string, color: string, icon: string }>
- `priorityConfig`: Record<Priority, { label: string, color: string, icon: string }>
- `typeConfig`: 覆盖 3 种类型 — Record<'requirement' | 'task' | 'bug', { label: string, color: string, icon: string }>
- `severityConfig`: Record<Severity, { label: string, color: string }>
- `statusOptions`: 用于下拉选择的选项数组（从 config 自动生成）

**project.ts 核心内容：**
- `projectStatusConfig`: 项目状态映射
- `projectTypeConfig`: 项目类型映射
- `moduleConfig` 类型定义 — `{ milestones: boolean, workflows: boolean, requirements: boolean }`。关闭里程碑模块时进入纯看板模式，关闭需求视图时进入纯任务管理模式（参考 PIVR 说明书 4.3 节）

**milestone.ts 核心内容：**
- `pivrPhases`: PIVR 四阶段的定义（code, label, description, color, icon）
- `milestoneTypeConfig`: 按项目类型映射每个 PIVR 阶段的含义（参考 PIVR 说明书第 3 节的场景映射表）
- `milestoneModeConfig`: 强约束/滚动/周期三种模式的配置

**navigation.ts 核心内容：**（详见 Step 5 菜单定义）
- `globalMenuItems`: 全局视图主菜单
- `projectMenuItems(projectId)`: 项目视图菜单
- `workItemFilterItems`: 工作项筛选菜单
- `projectUtilityItems(projectId)`: 项目底部管理菜单

**验证标准：** 所有页面的硬编码映射替换为 config 引用，grep 确认无残留。

- **Step 2b (~1.5h):** 复杂交互组件

```
│   └── WorkItemFilters.vue       # 筛选栏组件（状态/类型/优先级/指派人）
├── milestone/
│   ├── MilestoneCard.vue         # 里程碑卡片（展示 PIVR 阶段、进度条）
│   ├── MilestoneTimeline.vue     # PIVR 时间线视图（P→I→V→R 四阶段可视化）
│   ├── MilestoneWorkItems.vue    # 里程碑下的需求/任务列表（含进度 Roll-up 展示）
│   └── MilestoneCreateModal.vue  # 创建里程碑弹窗
```

**注意：** 不再有 `sprint/` 组件目录。

**PIVR 可视化重点：** `MilestoneTimeline.vue` 是 PIVR 模型在前端的核心表达。四个阶段展示为水平时间线，每个阶段有独立的颜色标识、进度百分比和当前状态标记。

**WorkItem 差异化策略：** `WorkItemCard.vue` 根据 `type` 字段渲染不同的 UI：
- `requirement`（需求）: 显示关联文档链接、评审状态、验收标准摘要、子项数量和完成进度
- `task`（任务）: 显示预估工时、实际工时、代码关联；若有 parent_id 则显示父任务引用（即子任务）
- `bug`（缺陷）: 显示严重程度、复现步骤摘要、关联需求

### Step 3: 可复用页面逻辑 Composables（~1h CC）

```
app/composables/
├── useWorkItemList.ts     # 工作项列表通用逻辑
│   # - 分页、搜索、筛选状态
│   # - 批量选择与批量操作
│   # - 排序切换
│   # - 从 config 自动生成筛选选项
│
├── useBoardDrag.ts        # 看板拖拽逻辑
│   # - 拖拽开始/结束/放置
│   # - 拖拽验证：初始化时从 GET /api/v1/projects/:id/workflow-transitions 预加载
│   #   响应格式：{ transitions: Array<{ fromStatus: string, toStatus: string, allowed: boolean, conditions?: string[] }> }
│   #   缓存当前项目的工作流规则（参考 workflow_transitions 表结构）
│   # - 乐观更新：先更新本地 store 状态，API 调用失败时回滚到拖拽前快照
│   # - 动画状态：拖拽中 → 验证中 → 成功/失败
│
├── useMilestoneProgress.ts # 里程碑进度计算
│   # - 服务端计算 Roll-up，客户端仅调用 API 展示（保证一致性）
│   # - API: GET /api/v1/milestones/:id/stats → { totalWeight, completedWeight, progress }
│   # - 服务端公式：阶段进度 = Σ(已完成需求权重) / Σ(总需求权重)
│   # - PIVR 四阶段独立计算，项目总进度 = 四阶段加权平均
│   # - 风险预警：逾期检测（当前日期 > deadline 且 progress < 100%）
│
├── useFormValidation.ts   # 表单验证工厂
│   # - 基于 Zod schema 生成验证函数
│   # - 统一错误消息格式
│   # - 与 Nuxt UI UForm 集成
│
└── useApiRequest.ts       # 通用 API 请求 composable
    # - 统一 loading/error/data 三态管理
    # - 自动重试策略（网络错误重试 1 次）
    # - 全局错误通知（toast 提示）
    # - 乐观更新辅助：snapshot() / rollback()
```

**useWorkItemList 核心 API：**
```typescript
const {
  items, loading, error,
  filters, search, sort,
  pagination,
  selectedIds, selectAll, toggleSelect,
  batchUpdate, batchDelete,
  refresh
} = useWorkItemList({
  projectId: string,
  defaultType?: WorkItemType,   // 可按类型预筛选
  defaultFilters?: Partial<Filters>
})
```

### Step 4: Zod 表单验证集成（~30 min CC）

创建 `app/schemas/` 目录：

```
app/schemas/
├── project.ts      # 项目创建/编辑 schema
├── work-item.ts    # 工作项创建/编辑 schema（按 type 差异化）
├── milestone.ts    # 里程碑 schema（含 PIVR 阶段校验）
```

**关键验证规则：**
- 项目名称：2-100 字符，不能纯空格
- 里程碑日期：结束日期必须晚于开始日期（强约束模式下必填）
- 工作项：必填字段（标题、类型、优先级）；Bug 类型必须有严重程度
- 里程碑必须关联 PIVR 阶段（phase 字段必须是 P/I/V/R 之一）

与 Nuxt UI 的 `UForm` 组件直接集成，传入 Zod schema 即可自动验证。
