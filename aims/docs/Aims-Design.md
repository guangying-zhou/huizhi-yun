## Problem Statement

汇智云平台已完成 Account（身份管理）和 Codocs（文档协作）两个基座模块，现在需要构建核心业务模块 Aims（研发项目管理），让平台从"文档协作工具"升级为"研发管理一体化平台"。

公司内部目前使用云效/teambition 等专业工具做项目管理，最大痛点是**多系统割裂、数据不通**——项目任务在一个系统、文档在另一个系统、审批在第三个系统、代码在 GitLab，无法串联。

## Demand Evidence

- 内部团队已经在使用 Account + Codocs，验证了平台基座的可行性
- PRD 明确了"一体化"为核心价值主张，Aims 是实现这一价值的关键拼图
- 多系统割裂是中小软件企业的普遍痛点（PRD 4.3 节），这不是假设而是真实的日常痛苦
- KPI 目标明确：任务按期完成率 ≥ 85%，用户跨模块使用率 ≥ 3 模块/用户

## Status Quo

当前状态：项目管理用云效/teambition，文档在 Codocs，代码在 GitLab，审批走单独流程。

这意味着：
- 项目经理在云效/teambition查任务进度，在 Codocs 写需求文档，在 GitLab 看代码，在微信催审批——每天在 4 个系统间切换
- 开发人员提交代码后手动回到云效/teambition更新任务状态，经常忘记
- "这个需求对应的任务做到哪了？代码合并了吗？"——回答这个问题需要跨 3 个系统查找

## Target User & Narrowest Wedge

**首要用户：** 公司内部研发团队（项目经理 + 开发人员）

**最小楔子（Narrowest Wedge）：** 不是做一个更好的云效/teambition，而是做一个**让任务、文档、代码在同一个界面流动**的工具。用户在 Aims 看到一个任务，点进去就能看到关联的需求文档（Codocs）、关联的代码提交（GitLab）、关联的审批状态——这是云效和teambition都做不到的。

## Constraints

1. **Workflow 模块尚在开发中**——Aims MVP 需使用审批功能时可同步测试 Workflow，
2. **Account 和 Codocs 已稳定**——可以安全依赖其 API（Codocs尚未对外提供，可随时提出需求即时实现）
3. **技术栈统一**——必须使用 Nuxt 4 + Nuxt UI V4 + MySQL，与其他模块保持一致
4. **端口 3002**——已在平台规划中分配
5. **内部团队先用**——MVP 面向的是已经在用云效/teambition的内部团队，必须在体验上有明显优势才能推动切换

## Premises

1. **一体化是核心卖点**——Aims MVP 必须从第一天就展示跨模块联动能力（任务↔文档、任务↔代码），否则它只是另一个云效/teambition
2. **内部团队是第一批用户**——需要在体验上说服他们从云效/teambition切换过来，迁移成本不低
3. **Account 和 Codocs 已稳定可依赖，Workflow 尚待一起测试**——审批相关功能利用Workflow
4. **"做得比云效/teambition好"不够**——差异化在于联动，不在于单点功能更优

## Approaches Considered

### Approach A: "最小可用"——任务看板为中心

- **Summary:** 只做项目 CRUD、任务看板、缺陷跟踪，快速上线验证
- **Effort:** S（人工 2 周 / CC 2 天）
- **Risk:** Low
- **Pros:**
  - 最快上线
  - 风险最低
- **Cons:**
  - 没有差异化，和云效/teambition比没有明显优势
  - 无法展示"一体化"卖点
  - 内部团队没有动力切换
- **Completeness:** 4/10

### Approach B: "一体化体验"——联动为中心 (CHOSEN)

- **Summary:** 在项目看板基础上，加入里程碑管理、GitLab 关联、Codocs 关联、企业微信通知、基础工时，展示一体化价值
- **Effort:** M（人工 4 周 / CC 4-5 天）
- **Risk:** Medium
- **Pros:**
  - 从第一天就展示跨模块联动——核心差异化
  - 里程碑管理覆盖了项目分阶段推进的日常需求
  - GitLab 关联让开发人员在一个界面看到任务+代码
  - 工作量可控，CC 加持下 4-5 天可完成
- **Cons:**
  - 比方案 A 复杂，需要与多个外部系统交互
  - GitLab API 集成可能有坑
  - 没有报表和仪表板，数据洞察能力弱
- **Reuses:** Account API（用户/部门/项目）、Codocs（文档链接）、企业微信通知（已有模板）
- **Completeness:** 7/10

### Approach C: "全量体验"——完整 MVP

- **Summary:** 方案 B + 需求管理 + 报表 + 仪表板 + 项目模板 + 版本管理
- **Effort:** L（人工 6-8 周 / CC 7-10 天）
- **Risk:** High
- **Pros:**
  - 最完整的 MVP，覆盖研发管理全流程
  - 有报表和仪表板，管理层也有使用价值
  - 上线即可全面替代禅道
- **Cons:**
  - 范围太大，延期风险高
  - 功能多但每个功能可能打磨不够
  - 用户反馈收集周期长
- **Completeness:** 9/10

## Recommended Approach

**选择方案 B："一体化体验"**

理由：Aims 的核心差异化不是"功能比云效/teambition多"，而是"数据在模块间流动"。方案 B 精准地展示了这个卖点：

- 在任务详情里看到关联的 Codocs 文档——**替代跨系统查文档**
- 在任务详情里看到关联的 GitLab commit——**替代手动更新任务状态**
- 任务指派/截止时收到企业微信通知——**替代微信催进度**
- 项目看板管理日常工作——**替代云效/teambition的日常协作**

方案 A 太薄，没有差异化动力；方案 C 太重，有延期风险。方案 B 是"刚好足够证明一体化价值"的范围。

### 技术设计要点

**0. 项目数据架构决策（已决定：三层驱动 + 项目集 + 独立项目主表）**

### 核心架构：三层驱动模型

参考《项目管理系统核心架构设计指导书V1.0》，采用**三层驱动架构**：

- **第一层：里程碑引擎（目标层）** — 项目的"锚点"，支持三种模式：
  - **强约束模式**：必须设截止日期，关联交付物检查清单（传统政务/大型项目）
  - **滚动计划模式**：完成即发布，无严格截止日期约束（SaaS/持续迭代）
  - **周期性模式**：月度/周度周期单元，如"2026年3月度维护"（运维/日常重复）
  - 同一项目内不同里程碑可使用不同模式（如交付阶段强约束 → 试运行滚动计划 → 正式运行周期维护）
- **第二层：需求看板（价值层）** — 通过 `work_items` 的 `type=requirement` 筛选展示
- **第三层：任务看板（执行层）** — 通过 `work_items` 的 `type=task` 筛选展示

**里程碑是工作项的一级容器**（`milestone_id` 必填），不再引入 Sprint 层级。

### 核心认知变更

Account 模块的 `projects` 表本质上是 **GitLab 仓库/组的注册表**，不是业务意义上的"研发项目"。Aims 管理的"项目"是有明确目标、由关联事项组成、追求投入产出比的业务活动。两者是 N:M 关系：

- 一个 Aims 项目（如"汇智云 v2.0"）可关联多个仓库
- 一个仓库可被多个 Aims 项目引用
- 有些 Aims 项目没有仓库（如交付实施、运维保障）

因此 Aims **建立独立的项目主表** `aims_projects`，而非 Account 的 1:1 扩展表。

### 项目集（Portfolio）

支持将一组项目统一管理，通过独立的 `project_portfolios` 表实现：
- 项目集有自己的属性（名称、描述、负责人、业务领域、部门）
- 项目通过 `portfolio_id` 外键归属项目集（1:N，一个项目最多归属一个项目集）
- 项目列表页按项目集分组展示（卡片视图嵌套、列表视图树状）

### 项目分类体系

```
Aims 项目分类
├── 🔵 技术类（MVP 支持）
│   ├── product_dev    — 产品研发：自研产品版本迭代（敏捷/Scrum）
│   ├── custom_dev     — 定制开发：为客户定制开发（敏捷+里程碑）
│   ├── delivery       — 交付实施：部署+培训+验收（瀑布/检查清单）
│   └── maintenance    — 运维保障：系统维护+故障处理（看板/工单驱动）
│
├── 🟡 经营类（Beta 预留）
│   ├── sales          — 销售项目：重大商机跟进
│   └── presales       — 售前项目：方案编写+POC
│
└── 🟢 管理类（GA 预留）
    ├── improvement    — 内部改进：流程优化、工具建设
    └── compliance     — 专项行动：安全整改、合规认证
```

**设计原则**：
- Account 继续管"平台注册"（仓库、组织、用户、业务领域），Aims 管"业务项目"
- Aims 项目通过 `aims_project_repos` 表 N:M 关联 Account 的仓库
- Aims 项目成员独立管理（`aims_project_members`），不再依赖 Account 的 project_members
- `dept_code`、`leader_uid`、`domain_code` 引用 Account 的部门、用户、业务领域数据
- `customer_code` 引用 Altoc 的客户编码
- 里程碑是工作项的一级容器

**1. 统一工作项模型（三层驱动）**
```
milestones 表 → 目标层（里程碑引擎，mode 区分强约束/滚动/周期）
work_items 表 → type 字段区分：requirement(价值层), task(执行层), bug(缺陷)
                milestone_id(NOT NULL) → 里程碑是一级容器
                parent_id → 支持嵌套（需求→任务→子任务）
                weight → 用于里程碑进度 Roll-up 计算
work_item_relations 表 → 关联关系：blocks, blocked_by, relates_to
```
好处：一套 API 管所有类型，需求看板和任务看板是同一数据模型的两个视图。里程碑进度 = Σ已完成工作项权重 / Σ总权重。

**1.1 工作项编号规则（已决定）**

采用**项目级显示编号**：`{project_code}-{seq}`，例如 `AIMS-1`, `MOBILE-2`。
- `project_code` 用于平台内唯一标识和跨模块关联，大写英文字母组合加数字
- `item_number` 为项目内自增序号（`work_items` 表中 project 级别的 AUTO_INCREMENT 或应用层生成）
- 权限矩阵中的角色权限已足够 MVP 使用，项目级粒度即可，不需要字段级权限

**2. 审批状态机（Workflow 不可用时的替代方案）**

状态枚举值统一使用英文，中文仅作 UI 展示标签：
```
Account 项目注册状态：enabled/disabled（平台级启停，不承载业务语义）
项目业务状态：draft(草稿) → approval_pending(待立项) → active(进行中) → paused(暂停) → completed(已完成) → archived(已归档)
里程碑：planning(规划中) → active(进行中) → completed(已完成，可reopen)
需求：draft(草稿) → reviewing(评审中) → confirmed(已确认) → developing(开发中) → completed(已完成，可reopen)
任务：todo(待办) → in_progress(进行中) → done(已完成，可reopen)
缺陷：new(新建) → confirmed(已确认) → fixing(修复中) → verifying(待验证) → closed(已关闭，可reopen)
```

项目业务状态和工作项状态都通过配置化状态转换表实现（`workflow_transitions` 表定义合法流转路径），便于后续 Workflow 模块接入时无缝替换。`aims_projects` 和 `work_items` 预留 `approval_status` 与 `workflow_instance_id` 字段。

**模块开关机制**（`aims_projects.module_config` JSON 字段）：
- `milestonesEnabled`：关闭后项目以纯看板模式运行（适用于纯运维/即时任务）
- `processAuditEnabled`：关闭后状态跳转无需审批（适用于敏捷团队）

**3. GitLab 集成策略**
- 通过 Account API（`GET /api/v1/projects/:id`）获取项目-GitLab 仓库映射关系，不直接访问 Account 数据库
- MVP 采用**手动触发同步 + 页面刷新按钮**（而非定时轮询），避免 GitLab API Rate Limit 问题和轮询复杂度
- 正则匹配 commit message 中的 `#{project_code}-\d+` 自动关联
- 存储 `last_synced_at` 和 `last_commit_sha` 用于增量拉取

**4. Codocs 关联策略**
- `project_documents` 统一承载工作项文档归属，工作项详情按 `project_documents.work_item_id` 查询
- MVP 只做 **Aims→Codocs 单向关联**：任务详情展示关联文档列表，点击跳转 Codocs
- Codocs 反向链接（文档详情展示关联的 Aims 任务）延后到 Phase 2，避免修改已稳定的 Codocs 模块

**5. 看板拖拽技术选型**
- 优先使用 `@formkit/drag-and-drop`（更轻量、原生 Vue 3 支持、无 jQuery 依赖）；若兼容性有问题则回退到 `vuedraggable-plus`

## 与资产管理系统（Assets）的对接规划

资产管理系统 PRD（`assets/docs/企业资产管理系统-PRD-V1.0.md`）明确了与 Aims 的关系。以下为对接点规划：

### MVP 阶段：无需对接，但数据模型预留友好

- `project_code` 作为跨模块唯一标识——资产系统的"关联项目"字段将使用同一个 `project_code`
- 工时数据（`time_entries` 表）设计为可导出——后续资产系统可按项目汇总人力成本

### Beta / GA 阶段：逐步对接

| 对接点 | 方向 | 说明 | 阶段 |
|--------|------|------|------|
| 项目关联资产 | Assets → Aims | Aims 项目详情展示"项目关联资产"列表（调 Assets API） | Beta |
| 项目成本汇总 | Assets + Aims → 报表 | 资源成本（Assets）+ 人力成本（Aims 工时）= 完整项目成本 | GA |
| 项目归档触发回收 | Aims → Assets | 项目状态变为 archived 时，通知资产系统触发资产回收任务 | GA |
| 产品版本关联研发项目 | Assets → Aims | 产品版本的"研发项目"字段关联 Aims 项目（Assets P2） | GA |
| 环境关联 | Assets ↔ Aims | Aims 发布/部署关联资产系统的环境视图 | GA |
| 数字资产关联 | Assets ↔ Aims | 代码仓库、交付物等数字资产关联到 Aims 项目/里程碑 | GA |

### 对接原则

- 通过 `project_code` 关联，不建立直接外键——两个系统独立数据库
- 对接通过 REST API 而非数据库直连
- 资产系统主动调 Aims API 获取工时数据，Aims 不推送
