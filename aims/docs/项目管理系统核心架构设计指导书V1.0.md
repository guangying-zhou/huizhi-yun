# 项目管理系统核心架构设计指导书 (V1.0)

## 1. 设计哲学：结构化灵活性 (Structured Flexibility)

在 AI 辅助开发背景下，系统设计的核心不再是单纯的任务记录，而是**价值流的确定性管理**。

* **管理层看“结果”**：通过里程碑锁定交付节点与合规性。
* **执行层看“效率”**：通过双视图看板利用 AI 快速拆解并完成任务。

***

## 2. 核心逻辑模型：三层驱动架构

## 2.1 第一层：里程碑引擎 (The Goal Layer)

里程碑是项目的“锚点”，支持三种配置模式以适配不同客户：

1. **强约束模式（传统政务/大型项目）**：
   * 必须设置截止日期。
   * 关联交付物检查清单（如：需求说明书、测试报告）。
   * **AI 介入点**：自动检查交付物完整性，根据往期进度预警逾期风险。
2. **滚动计划模式（SaaS/持续迭代）**：
   * 里程碑可不设截止日期（即“逻辑版本”）。
   * 采用“完成即发布”逻辑。
3. **周期性/维护模式（运维/日常重复型）**：
   * **统一逻辑**：将里程碑抽象为“周期单元”（如：2026年3月度维护里程碑）。
   * 系统支持自动创建月度/周度里程碑，简化初始化操作。

## 2.2 第二层：需求看板视图 (The Value Layer)

* **对象**：用户故事 (User Stories) 或 功能特性 (Features)。
* **状态流动**：待处理 -> 细化中 -> 开发中 -> 测试中 -> 已发布。
* **核心功能**：一个需求卡片可穿透查看其关联的所有子任务进度。

## 2.3 第三层：任务看板视图 (The Execution Layer)

* **对象**：原子任务 (Tasks)，如“编写接口”、“设计UI”。
* **AI 集成**：支持“**一键拆解**”。AI 根据需求描述，自动生成技术任务列表建议，由开发人员勾选确认后进入看板。

***

## 3. 关键功能模块设计

## 3.1 双视图联动逻辑

系统必须实现**数据的自动向上卷起 (Roll-up)**：

* **状态联动**：当某个需求下的所有任务卡片进入“Done”状态时，需求卡片自动标记为“待验收”。
* **进度比例**：里程碑进度 = \$\frac{\sum 已完成需求权重}{\sum 总需求权重}\$。

## 3.2 AI 进度预测功能 (Killer Feature)

利用 AI 学习团队的**速率 (Velocity)**：

* **预测逻辑**：AI 分析过去 3 个月内，团队从“需求入池”到“任务关闭”的平均周期。
* **风险预警**：若当前活跃任务的复杂度超过了剩余时间所能支撑的速率，里程碑进度条变色警告。
* **动态调整建议**：AI 建议“若要按时交付里程碑 A，需将优先级低的需求 B 移至下一里程碑”。

## 3.3 模块开关机制 (Module Toggle)

为了解决“系统太重”的问题，项目创建时支持：

* **\[开关] 里程碑模块**：关闭后，项目直接以单一看板模式运行（适用于纯运维、即时性任务）。
* **\[开关] 流程审计**：关闭后，任务状态跳转无需审批（适用于敏捷团队）。

## 4. 目标客户场景适配表

| **客户类型**    | **推荐模式配置**       | **核心依赖工具**           |
| :---------- | :--------------- | :------------------- |
| **智慧政务**    | 强约束里程碑 + 双视图看板   | 甘特图、基准对比、PDF周报导出     |
| **SaaS 研发** | 滚动计划里程碑 + 需求看板驱动 | 代码库集成 (GitLab)、版本号管理 |
| **互联网项目**   | 极简看板（隐藏里程碑）      | CI/CD 流水线状态、实时燃尽图    |
| **日常维护**    | 月度自动里程碑 + 任务看板   | 循环任务模版、工时统计          |
