## Problem Statement

Aims 前端 MVP 功能已基本实现（项目管理、迭代、工作项、缺陷、看板），但代码组织混乱导致后续开发效率低下。核心问题：重复常量散布在 5+ 个页面、只有 6 个组件（所有业务逻辑塞在页面里）、PIVR 三层架构在前端缺乏统一表达、表单验证缺失、布局文件过重。

## Demand Evidence

- 开发团队和管理层都在推动 AIMS 尽快上线替代禅道
- 当前每新增一个页面就要重新定义状态映射、颜色映射、图标映射，开发效率低
- PIVR 模型（Preparation→Implementation→Validation→Refinement）是产品核心差异化，但前端没有清晰的组件对应
- Beta 阶段需要增加甘特图、报表、需求管理等大量页面，当前架构无法高效支撑

## Status Quo

### 当前前端状态

**已实现的核心功能：**
- 23 个页面（其中约 12 个功能完善，11 个仅脚手架）
- 6 个 Pinia stores（project, workItem, sprint, milestone, account, portfolio）
- 9 个 composables
- 6 个组件（极少）
- 3 个类型定义文件

**具体问题清单：**

| 问题 | 影响范围 | 严重程度 |
|------|---------|---------|
| 状态/优先级/类型的 label/color/icon 映射在 5+ 个页面各写一遍 | work-items, tasks, bugs, board, plan 等 | 高 |
| 无共享业务组件，业务逻辑全在页面 script 里 | 所有功能页面 | 高 |
| WorkItem 按 type（需求/任务/Bug）的 UI 差异化无统一策略 | work-items, tasks, bugs | 高 |
| default.vue 布局 400+ 行，混入项目集树构建逻辑 | 全局 | 中 |
| 表单验证零散，未使用 Zod | 所有表单操作 | 中 |
| 看板拖拽 drop handler 未完成 | board.vue | 中 |
| API 请求/响应类型部分不明确（nullable 字段、partial 对象） | 全局 | 中 |

## Constraints

1. **不能中断现有功能** — 重构过程中所有已实现功能必须继续可用
2. **技术栈锁定** — Nuxt 4 + Vue 3 + Nuxt UI V4 + Pinia + Tailwind CSS
3. **必须符合 PIVR 模型** — 里程碑固定 PIVR 四阶段，前端组件必须体现这一设计
4. **时间约束** — MVP 目标 2026 年 4 月中旬，重构不能显著推迟交付

## Premises

1. 核心问题不是功能缺失，而是架构混乱——"胖页面"模式是最大债务
2. PIVR 三层架构（里程碑→需求→任务）需要在前端有清晰的组件和数据流对应
3. 重构必须在继续开发新功能之前完成，否则技术债会继续累积
4. 目标是让新页面开发效率翻倍——开发者不需要再从头写状态映射和列表逻辑

## Approaches Considered

### Approach A: 常量+组件提取（Quick Win）
只做最小重构——提取重复常量和 3-5 个常用 UI 片段为组件。不改页面结构。
- Effort: S | Risk: Low | Completeness: 4/10
- Pros: 立即消除重复，风险最低
- Cons: "胖页面"依然存在，治标不治本

### Approach B: 前端架构标准化（CHOSEN）
系统性重构：共享常量、业务组件库、可复用 composables、Zod 验证、布局瘦身。
- Effort: M | Risk: Med | Completeness: 8/10
- Pros: 从根本解决架构问题，新功能开发效率翻倍
- Cons: 需要 4-6 小时集中重构，可能引入 regression

### Approach C: 全面组件化+设计系统
在 B 基础上加 Storybook、完整类型系统、E2E 测试。
- Effort: L | Risk: Med-High | Completeness: 10/10
- Pros: 生产级质量
- Cons: 对内部工具过度工程化

## Recommended Approach

### 方案 B：前端架构标准化

分 6 个步骤执行，每步独立可验证：

---

### Step 1: 共享常量与枚举配置（~30 min CC）

创建 `app/config/` 目录，集中管理所有业务常量：

```
app/config/
├── work-item.ts      # WorkItem 相关的所有映射
├── project.ts        # 项目状态、类型映射
├── milestone.ts      # PIVR 阶段定义、颜色、图标
└── index.ts          # 统一导出
```

**work-item.ts 核心内容：**
- `statusConfig`: Record<WorkItemStatus, { label: string, color: string, icon: string }>
- `priorityConfig`: Record<Priority, { label: string, color: string, icon: string }>
- `typeConfig`: 覆盖全部 5 种类型 — Record<'epic' | 'story' | 'task' | 'sub_task' | 'bug', { label: string, color: string, icon: string }>
- `severityConfig`: Record<Severity, { label: string, color: string }>
- `statusOptions`: 用于下拉选择的选项数组（从 config 自动生成）

**project.ts 补充内容：**
- `moduleConfig`: 模块开关类型定义 — `{ milestones: boolean, workflows: boolean, requirements: boolean }`。关闭里程碑模块时进入纯看板模式，关闭需求视图时进入纯任务管理模式（参考 PIVR 说明书 4.3 节）。所有依赖里程碑的组件和导航菜单项需根据此配置做条件渲染。

**milestone.ts 核心内容：**
- `pivrPhases`: PIVR 四阶段的定义（code, label, description, color, icon）
- `milestoneTypeConfig`: 按项目类型映射每个 PIVR 阶段的含义（参考 PIVR 说明书的场景映射表）
- `milestoneModeConfig`: 强约束/滚动/周期三种模式的配置

**验证标准：** 所有页面的硬编码映射替换为 config 引用，grep 确认无残留。

---

### Step 2: 业务组件库建设（~2h CC）

分两个子步骤：
- **Step 2a (~30min):** Badge/简单展示组件（StatusBadge, PriorityBadge, TypeBadge, UserAvatar, EmptyState）
- **Step 2b (~1.5h):** 复杂交互组件（DetailSideover, CreateModal, Filters, KanbanColumn, MilestoneTimeline）

按业务域创建组件：

```
app/components/
├── work-item/
│   ├── WorkItemCard.vue          # 看板卡片（支持需求/任务/Bug 差异化展示）
│   ├── WorkItemStatusBadge.vue   # 状态标签（颜色、图标来自 config）
│   ├── WorkItemPriorityBadge.vue # 优先级标签
│   ├── WorkItemTypeBadge.vue     # 类型标签
│   ├── WorkItemDetailSideover.vue # 工作项详情侧边栏（从页面提取）
│   ├── WorkItemCreateModal.vue   # 创建工作项弹窗（Zod 验证）
│   └── WorkItemFilters.vue       # 筛选栏组件（状态/类型/优先级/指派人）
├── milestone/
│   ├── MilestoneCard.vue         # 里程碑卡片（展示 PIVR 阶段、进度条）
│   ├── MilestoneTimeline.vue     # PIVR 时间线视图（P→I→V→R 四阶段可视化）
│   ├── MilestoneWorkItems.vue    # 里程碑下的需求/任务列表（含进度 Roll-up 展示）
│   └── MilestoneCreateModal.vue  # 创建里程碑弹窗
├── board/
│   ├── KanbanColumn.vue          # 看板列
│   ├── KanbanCard.vue            # 看板卡片（WorkItemCard 的薄包装，加拖拽 handler 和拖拽状态样式）
│   └── KanbanSwimLane.vue        # 泳道
├── sprint/
│   ├── SprintCard.vue            # 迭代卡片
│   └── SprintCreateModal.vue     # 创建迭代弹窗
├── project/
│   └── ProjectCard.vue           # 项目卡片（项目列表页使用，含 PIVR 进度概览）
└── common/
    ├── UserAvatar.vue            # 用户头像（复用 useAvatar）
    ├── DateRangePicker.vue       # 日期范围选择器
    └── EmptyState.vue            # 空状态占位
```

**PIVR 可视化重点：** `MilestoneTimeline.vue` 是 PIVR 模型在前端的核心表达。它将四个阶段展示为水平时间线，每个阶段有独立的颜色标识、进度百分比和当前状态标记。技术管理者打开项目概览页就能看到项目处于 PIVR 的哪个阶段。

**WorkItem 差异化策略：** `WorkItemCard.vue` 根据 `type` 字段渲染不同的 UI：
- `requirement`: 显示关联文档链接、评审状态、验收标准摘要
- `task`: 显示预估工时、实际工时、代码关联
- `sub_task`: 精简卡片，显示父任务引用、完成状态
- `bug`: 显示严重程度、复现步骤摘要

### Step 3: 可复用页面逻辑 Composables（~1h CC）

提取页面中重复的逻辑为 composables：

```
app/composables/
├── useWorkItemList.ts     # 工作项列表通用逻辑
│   # - 分页、搜索、筛选状态
│   # - 批量选择与批量操作
│   # - 排序切换
│   # - 从 config 自动生成筛选选项
│
├── useBoardDrag.ts        # 看板拖拽逻辑
│   # - 拖拽开始/结束/放置
│   # - 拖拽验证：初始化时从 GET /api/v1/projects/:id/workflow-transitions 预加载
│   #   响应格式：{ transitions: Array<{ fromStatus: string, toStatus: string, allowed: boolean, conditions?: string[] }> }
│   #   缓存当前项目的工作流规则（参考 workflow_transitions 表结构）
│   # - 乐观更新：先更新本地 store 状态，API 调用失败时回滚到拖拽前快照
│   # - 动画状态：拖拽中 → 验证中 → 成功/失败
│
├── useMilestoneProgress.ts # 里程碑进度计算
│   # - 服务端计算 Roll-up，客户端仅调用 API 展示（保证一致性）
│   # - API: GET /api/v1/milestones/:id/stats → { totalWeight, completedWeight, progress }
│   # - 服务端公式：阶段进度 = Σ(已完成需求权重) / Σ(总需求权重)
│   # - PIVR 四阶段独立计算，项目总进度 = 四阶段加权平均
│   # - 风险预警：逾期检测（当前日期 > deadline 且 progress < 100%）
│
├── useFormValidation.ts   # 表单验证工厂
│   # - 基于 Zod schema 生成验证函数
│   # - 统一错误消息格式
│   # - 与 Nuxt UI form 集成
│
└── useApiRequest.ts       # 通用 API 请求 composable
    # - 统一 loading/error/data 三态管理
    # - 自动重试策略（网络错误重试 1 次）
    # - 全局错误通知（toast 提示）
    # - 乐观更新辅助：snapshot() / rollback()
```

**useWorkItemList 核心 API：**
```typescript
const {
  items, loading, error,
  filters, search, sort,
  pagination,
  selectedIds, selectAll, toggleSelect,
  batchUpdate, batchDelete,
  refresh
} = useWorkItemList({
  projectId: string,
  defaultType?: WorkItemType,   // 可按类型预筛选
  defaultFilters?: Partial<Filters>
})
```

### Step 4: Zod 表单验证集成（~30 min CC）

创建 `app/schemas/` 目录：

```
app/schemas/
├── project.ts      # 项目创建/编辑 schema
├── work-item.ts    # 工作项创建/编辑 schema（按 type 差异化）
├── milestone.ts    # 里程碑 schema（含 PIVR 阶段校验）
├── sprint.ts       # 迭代 schema（日期范围校验）
└── index.ts
```

**关键验证规则：**
- 项目名称：2-100 字符，不能纯空格
- 里程碑日期：结束日期必须晚于开始日期
- 工作项：必填字段（标题、类型、优先级）；Bug 类型必须有严重程度
- 迭代日期：不能与同项目其他活跃迭代重叠

与 Nuxt UI 的 `UForm` 组件直接集成，传入 Zod schema 即可自动验证。

### Step 6: 页面瘦身与组件替换（~2h CC）

对已有的核心页面进行瘦身，替换为新组件和 composables：

**决策：tasks.vue 与 work-items.vue 合并。** 两个页面功能高度重叠（列表展示、筛选、批量操作、详情侧边栏），差异仅为默认筛选条件（tasks 只看 task 类型，work-items 看全部）。合并为 `work-items.vue`，通过 URL query 参数 `?type=task` 控制默认筛选。删除 `tasks.vue`。

**页面重构迁移表：**

| 页面 | Before（当前） | After（重构后） |
|------|-------------|--------------|
| **work-items.vue** | 内联 statusLabel/colorMap/iconMap（~50行）、内联筛选栏（~80行）、内联详情逻辑（~100行） | `<WorkItemFilters>` + `useWorkItemList()` + `<WorkItemDetailSideover>` ，script < 80行 |
| **tasks.vue** | 与 work-items 90% 重复 | **删除**，合并到 work-items.vue，通过 `?type=task` 预筛 |
| **bugs.vue** | 内联严重程度映射、重复的列表逻辑 | `useWorkItemList({ defaultType: 'bug' })` + severity 筛选 |
| **board.vue** | 内联拖拽状态（dragValidationToken 等）、未完成的 drop handler | `useBoardDrag()` + `<KanbanColumn>` + `<KanbanCard>` |
| **plan.vue** | 内联里程碑 CRUD、无 PIVR 可视化 | `<MilestoneTimeline>` + `<MilestoneCard>` + `<MilestoneWorkItems>` |
| **projects/index.vue** | 内联项目卡片渲染 | 提取 `<ProjectCard>` 组件 |

**模块开关适配：** 侧边栏和页面导航需根据 `project.moduleConfig` 条件渲染：
- `moduleConfig.milestones = false` → 隐藏 plan.vue 菜单项，board.vue 不显示里程碑分组
- `moduleConfig.requirements = false` → work-items.vue 隐藏需求类型筛选

**验证标准：** 每个页面瘦身后，核心逻辑应 < 150 行（不含 template），所有业务映射来自 config。
