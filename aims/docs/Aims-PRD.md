## 1. 概述

Aims（汇智云·项目）是汇智云（Huizhi.Yun）企业作业与管理云平台的核心业务模块，定位为**目标驱动的研发项目全生命周期管理平台**。

Aims 覆盖从项目立项到交付验收的完整研发管理链路：**立项 → 需求 → 迭代 → 任务 → 开发 → 测试 → 发布**，帮助软件研发团队实现项目过程可视化、任务协作高效化、质量度量数据化。

### 1.1 核心价值

- **全链路追溯**：需求 → 任务 → 代码提交 → 测试用例，每个环节可追溯
- **数据驱动决策**：燃尽图、迭代速度、缺陷趋势等度量指标辅助管理决策
- **AI 增效**：AI 辅助需求拆解、WBS 分解、智能排期，降低管理成本
- **平台一体化**：与 Account（身份）、Codocs（文档）、Workflow（审批）、Altoc（经营）深度联动

### 1.2 目标用户

| 角色 | 核心诉求 |
|------|---------|
| 项目经理 | 项目进度可视化，资源协调，风险预警 |
| 产品经理 | 需求管理，PRD 协作，需求追溯 |
| 开发人员 | 任务领取，代码关联，工时记录 |
| 测试人员 | 缺陷跟踪，测试管理，回归追踪 |
| 团队负责人 | 团队工作量分析，迭代健康度，交付质量 |
| 企业管理者 | 多项目全局视图，资源利用率，经营数据 |

## 2. 平台定位与模块关系

### 2.1 在汇智云中的位置

```
Account（身份中心）── 用户/部门/角色/权限/项目（所有模块共享）
     │
     ├── Codocs（文档）←→ Aims（项目）：设计文档关联任务，需求文档关联迭代，项目 Wiki
     │
     ├── Workflow（流程引擎）←→ Aims（项目）：立项审批、需求变更审批、发布审批
     │
     ├── Aims（项目）←→ Altoc（营销）：项目工时→成本核算，合同交付物→里程碑，运维工单→缺陷
     │
     └── Align（协同）←→ Aims（项目）：OKR 关联项目进度
```

### 2.2 模块间数据依赖

| 依赖模块 | Aims 使用的能力 | 接口方式 |
|----------|----------------|---------|
| Account | 用户信息、部门、角色、权限、GitLab 项目同步、**业务领域字典** | REST API（HZY_ACCOUNT_API） |
| Workflow | 审批流程发起、审批状态查询、审批回调 | REST API |
| Codocs | 文档关联、Wiki 链接 | 数据库关联 + 跳转链接 |
| Altoc | 工时数据输出、里程碑数据输出、**客户编码** | REST API（Altoc 调用 Aims） |

## 3. 功能需求

### 3.1 项目管理

#### 3.1.1 项目 CRUD

- 项目创建：名称、简称（`short_name`，必填）、代号、内部代号（`internal_code`，选填）、描述、负责人、起止日期、项目类型（敏捷/瀑布/混合）
- 项目编辑：基本信息变更
- 项目业务状态流转：草稿 → 待立项 → 进行中 → 暂停 → 已完成 → 已归档
- Account 的项目启停状态仅用于平台级可见性，不替代 Aims 业务状态
- 项目关闭与归档
- 项目列表：卡片视图 + 列表视图，支持筛选、搜索、排序

#### 3.1.2 项目立项审批

- 创建项目后进入"待立项"状态，可发起立项审批（Workflow 可用时对接）
- 审批通过后项目状态自动变更为"进行中"
- 审批驳回后可修改重新提交

#### 3.1.3 项目成员管理

- 从 Account 拉取用户，添加/移除项目成员
- 项目角色分配：项目经理、开发、测试、产品、观察者
- 成员权限：基于项目角色控制功能访问

#### 3.1.4 项目仪表板

- 项目进度概览：迭代进度、任务完成率、缺陷趋势
- 风险预警：逾期任务、阻塞任务、高优先级未处理缺陷
- 近期活动流：最近的任务变更、代码提交、缺陷更新
- 成员工作量分布

#### 3.1.5 项目模板

- 预置模板：敏捷开发模板（含迭代、看板、标准工作流）、瀑布模板（含阶段、里程碑、评审节点）
- 自定义模板：保存当前项目配置为模板
- 基于模板创建项目：自动初始化迭代规划、任务类型、工作流、看板配置

#### 3.1.6 版本管理

- 版本号管理（语义化版本）
- 版本规划：关联迭代和任务
- 发布日期管理
- Changelog 自动生成（基于关联的任务和缺陷）

### 3.2 需求管理

#### 3.2.1 需求 CRUD

- 需求创建：标题、描述、类型（功能需求/非功能需求/技术需求）、优先级、来源
- 需求详情：富文本描述（支持 Markdown）
- 需求状态流转：草稿 → 评审中 → 已确认 → 开发中 → 已完成 → 已关闭
- 需求列表：表格视图，支持筛选和搜索

#### 3.2.2 需求评审

- 发起需求评审，指定评审人
- 评审意见记录
- 评审通过/驳回/需修改

#### 3.2.3 需求变更

- 变更申请（对接 Workflow 审批流）
- 变更记录追踪：变更原因、影响范围、审批记录
- 变更历史对比

#### 3.2.4 需求追溯

- 需求 → 任务：一个需求拆分为多个任务
- 需求 → 代码：通过任务关联的 GitLab commit/MR
- 需求 → 测试用例：关联的测试用例及执行结果
- 追溯矩阵视图

#### 3.2.5 需求与 Codocs 关联

- 需求关联 Codocs 文档（PRD、设计文档）
- 从需求详情跳转到关联文档
- AI 辅助需求拆解：基于 PRD 文档自动建议需求拆分

### 3.3 迭代管理

> **设计变更：** 迭代(Sprint)已废弃，PIVR 里程碑作为工作项的唯一一级容器。原迭代功能由里程碑的滚动计划模式替代。

#### 3.3.1 迭代 CRUD

- 迭代创建：名称、目标、起止日期、关联版本
- 迭代规划：从 Backlog 拖拽任务到迭代
- 迭代容量规划：基于团队人数和工时估算
- 迭代状态：规划中 → 进行中 → 评审 → 已完成

#### 3.3.2 迭代看板

- Kanban 视图：待办 → 进行中 → 待测试 → 已完成（可自定义列）
- WIP 限制：限制每列最大任务数
- 泳道：按指派人、优先级、任务类型分组
- 拖拽操作：拖拽任务变更状态

#### 3.3.3 迭代燃尽图

- 理想燃尽线 vs 实际燃尽线
- 按故事点或任务数统计
- 范围变更标注

#### 3.3.4 迭代评审与回顾

- 迭代评审记录：完成情况、演示结果
- 迭代回顾：做得好的、待改进的、行动项
- 迭代速度趋势（跨迭代对比）

### 3.4 任务管理

#### 3.4.1 工作项类型

采用三种工作项类型：

- **requirement（需求）**：价值层，描述业务需求或功能需求
- **task（任务）**：执行层，具体的开发/测试/设计工作，支持通过 `parent_id` 嵌套实现子任务
- **bug（缺陷）**：质量层，缺陷跟踪与修复

每个工作项有 `tier` 字段区分粒度：
- **target（目标）**：待分解的高层工作项，不可直接指派
- **matter（事项）**：可直接指派给成员的具体工作项

#### 3.4.2 任务 CRUD

- 任务创建：标题、描述、类型、优先级（P0-P3）、指派人、预估工时
- 任务详情：富文本描述、附件、评论、操作日志
- 任务状态流转：待办 → 进行中 → 待审查 → 待测试 → 已完成（可自定义工作流）
- 批量操作：批量修改状态、指派人、优先级

#### 3.4.3 任务依赖

- 依赖关系类型：阻塞、被阻塞、关联
- 甘特图视图：展示任务时间线和依赖关系
- 关键路径标识

#### 3.4.4 AI 驱动的 WBS 分解

- 输入 Epic/Story 描述，AI 自动建议子任务拆分
- 自动估算工时
- 用户确认后批量创建

#### 3.4.5 智能排期与资源分配

- 基于成员可用工时和任务优先级自动排期
- 资源冲突检测与告警
- 排期建议（AI 辅助）

#### 3.4.6 任务视图

- **看板视图**：拖拽式 Kanban
- **列表视图**：表格展示，支持排序、筛选
- **甘特图视图**：时间线展示，依赖关系可视化
- **日历视图**：按日期展示任务

### 3.5 缺陷管理

#### 3.5.1 缺陷 CRUD

- 缺陷创建：标题、描述、复现步骤、严重程度（致命/严重/一般/建议）、优先级、指派人
- 缺陷截图/附件上传（OSS 存储）
- 缺陷状态流转：新建 → 已确认 → 修复中 → 待验证 → 已关闭（可重新打开）
- 缺陷列表：表格视图，支持多维度筛选

#### 3.5.2 缺陷关联

- 关联需求：缺陷来源于哪个需求
- 关联任务：修复该缺陷的开发任务
- 关联代码：修复缺陷的 commit/MR

#### 3.5.3 缺陷统计

- 缺陷趋势图：新增 vs 关闭
- 按严重程度分布
- 按模块分布
- 平均修复时长
- 缺陷重开率

### 3.6 代码管理（GitLab 集成）

#### 3.6.1 仓库关联

- 项目关联 GitLab 仓库（基于 Account 模块已有的 GitLab 项目同步）
- 仓库基本信息展示

#### 3.6.2 代码提交关联

- 通过 commit message 中的任务编号自动关联（如 `#AIMS-123`，前缀来自项目 `item_key_prefix`）
- 任务详情展示关联的 commit 列表
- commit 详情跳转 GitLab

#### 3.6.3 合并请求（MR）

- MR 列表展示
- MR 与任务关联
- MR 状态同步（开放/已合并/已关闭）
- 代码审查流程：评审人分配、评审意见

#### 3.6.4 代码质量看板

- 静态分析结果展示（对接 SonarQube 等工具，远期）
- 代码贡献统计（基于 nuxt-template 模块能力扩展）

### 3.7 测试管理

#### 3.7.1 测试计划

- 测试计划创建：名称、范围、关联迭代/版本
- 测试计划进度追踪

#### 3.7.2 测试用例

- 测试用例创建：标题、前置条件、测试步骤、预期结果
- 用例分类：功能测试、接口测试、性能测试、兼容性测试
- 用例与需求关联

#### 3.7.3 测试执行

- 测试执行记录：通过/失败/阻塞/跳过
- 失败用例自动创建缺陷
- 测试报告生成

### 3.8 工时管理

#### 3.8.1 工时记录

- 按任务记录实际工时
- 工时日志：日期、工时、工作描述
- 快捷记录：从任务详情直接录入

#### 3.8.2 工时统计

- 个人工时汇总（按日/周/月）
- 项目工时汇总
- 预估工时 vs 实际工时对比
- 工时数据输出给 Altoc（项目成本核算）

### 3.9 通知与提醒

#### 3.9.1 通知触发规则

- 任务指派/变更通知
- 任务截止日期提醒（提前 1 天、当天、逾期）
- 缺陷状态变更通知
- 迭代开始/结束提醒
- 审批结果通知
- @提及通知（评论中 @某人）

#### 3.9.2 通知渠道

- 站内通知（已有 NotificationsSlideover 组件）
- 企业微信推送（已有 wecom 配置）
- 测试模式：通知重定向到指定用户（NOTIFY_REDIRECT_TO）

### 3.10 报表与度量

#### 3.10.1 项目级报表

- 项目进度概览
- 里程碑燃尽图
- 里程碑交付趋势
- 缺陷趋势分析

#### 3.10.2 团队级报表

- 成员任务完成率
- 成员工时统计
- 工作量分布热力图

#### 3.10.3 组织级报表

- 多项目进度对比
- 资源利用率分析
- 代码贡献分析

#### 3.10.4 AI 洞察报表

- 项目健康度评分
- 风险预测：基于历史数据预测延期风险
- 效率建议：识别瓶颈，给出改进建议

### 3.11 交付物管理

#### 3.11.1 交付物定义

- `deliverables` 表管理各实体（project/milestone/work_item）的验收标准
- 交付物类型：document（文档）、code（代码）、artifact（制品）、task（事务）
- 模板自动生成：根据项目类别 + PIVR 阶段自动创建交付物模板
- 状态流转：pending（待提交）→ submitted（已提交）→ approved（已通过）/ rejected（已驳回）

#### 3.11.2 交付物关联

- 交付物可关联到项目、里程碑、工作项等不同层级
- 支持附件上传（OSS 存储）
- 交付物完成度作为里程碑/项目完成率的计算依据之一

### 3.12 统一审核

#### 3.12.1 审核机制

- `approval_records` 表统一管理所有状态流转的审核记录
- 覆盖范围：项目立项审批、里程碑完成确认、交付物评审
- 审核人自动推荐（项目集负责人），支持手动覆盖指定

#### 3.12.2 审核流程

- 发起审核时自动推荐审核人
- 审核意见记录（通过/驳回 + 备注）
- 审核结果触发对应实体状态变更
- 审核历史可追溯

### 3.13 项目文档管理

#### 3.13.1 文档结构

- `project_documents` 表替代原有的 `work_item_documents`
- 支持所有层级：项目集(portfolio)/项目(project)/里程碑(milestone)/工作项(work_item)
- 文件夹结构：通过 `parent_id` 嵌套实现目录树

#### 3.13.2 Codocs 集成

- 通过内嵌编辑器（iframe）集成 Codocs 模块
- 支持在 Aims 内直接创建和编辑协作文档
- OSS 存储路径规范：`aims-docs/{portfolio_code}/{project_code}/{pivr_stage}/{item_key}/`

#### 3.13.3 文档关联

- 文档可关联到项目集、项目、里程碑、工作项等实体
- 支持从文档详情跳转到关联的实体
- 文档版本管理（由 Codocs 协作层提供）

## 4. 非功能需求

### 4.1 性能

- 页面加载时间 ≤ 2 秒
- 看板拖拽操作响应 ≤ 200ms
- 支持单项目 10,000+ 任务
- API 响应时间 ≤ 500ms（P95）

### 4.2 安全

- CAS 单点登录 / 企业微信 OAuth
- 基于角色的访问控制（RBAC），复用 Account 权限体系
- 项目级数据隔离
- API 密钥认证（模块间调用）

### 4.3 可用性

- 响应式设计，支持桌面端和平板端
- 支持键盘快捷键（任务创建、状态变更等常用操作）
- 支持深色模式（基于 Nuxt UI V4）

### 5.2 数据库设计

数据库名：`hzy_aims`

核心表设计见 `aims_schema.sql`。

说明：
- Account 模块继续作为平台项目注册表，保存 `projects`、`project_members`、`business_domains`
- Aims 模块保存项目集、项目、里程碑、工作项、工时、GitLab 关联等研发管理数据
- `project_code` 作为跨模块唯一标识，可包含 `/`
- `project_code` 作为工作项展示编号与 Git 提交关联前缀，必须是全局唯一的大写短码
- 采用三层驱动架构：里程碑(目标层) → 需求(价值层) → 任务(执行层)
- 里程碑是工作项的一级容器（`milestone_id` 必填），不再引入 Sprint 层级

核心实体：
- `project_portfolios` — 项目集（项目组合管理）
- `aims_projects` — 项目主表（含 portfolio_id、domain_code、customer_code、module_config）
- `aims_project_members` — 项目成员
- `aims_project_repos` — 项目-仓库关联（N:M）
- `project_counters` — 项目内工作项计数器
- `milestones` — 里程碑（目标层，mode: strong_constraint/rolling_plan/periodic）
- `work_items` — 工作项（统一存储 requirement/task/bug，milestone_id 必填，支持 parent_id 嵌套，weight 用于进度 Roll-up）
- `work_item_relations` — 工作项关联关系（依赖、阻塞）
- `work_item_comments` — 工作项评论
- `work_item_attachments` — 工作项附件
- `work_item_changelog` — 工作项变更日志
- `workflow_transitions` — 工作流流转规则（支持 milestone 实体类型）
- `time_entries` — 工时记录
- `gitlab_commits` — GitLab 提交关联
- `notification_rules` — 通知规则
- `project_documents` — 项目文档（替代 work_item_documents，支持 portfolio/project/milestone/work_item 所有层级，parent_id 嵌套目录树）
- `deliverables` — 交付物（验收标准管理，关联 project/milestone/work_item）
- `approval_records` — 统一审核记录（项目审批、里程碑完成确认、交付物评审）

> **注意：** `work_item_documents` 已废弃，由 `project_documents` 替代。

Beta / GA 阶段再扩展：`versions`、`test_plans`、`test_cases`、`test_executions`、`gitlab_merge_requests` 等。

### 5.3 项目结构

```
aims/
├── app/
│   ├── components/
│   │   ├── project/          # 项目相关组件
│   │   ├── board/            # 看板组件
│   │   ├── task/             # 任务组件
```

## 7. 关键设计决策

### 7.1 统一工作项模型

采用统一的 `work_items` 表存储 requirement（需求）、task（任务）、bug（缺陷）三种工作项类型，通过 `type` 字段区分，`tier` 字段区分粒度（target/matter）。优势：

- 简化 API 设计（统一 CRUD）
- 关联关系灵活（任意类型间可建立关联）
- 看板、列表等视图通过筛选 type 实现
- task 类型通过 parent_id 嵌套支持子任务，无需独立的 sub-task 类型

### 7.2 可自定义工作流

工作流不硬编码，通过 `workflows` + `workflow_transitions` 表定义状态和流转规则：

- 每个项目可选择不同的工作流
- 管理员可自定义状态和流转条件
- 预置"敏捷开发"和"瀑布开发"两套工作流

### 7.3 Workflow 模块集成策略

审批类流程（立项、需求变更、发布审批）统一走 Workflow 模块，Aims 不自建审批引擎：

- 调用 `POST /api/v1/instances/prepare` 准备发起
- 调用 `POST /api/v1/instances` 正式发起
- 通过回调接收审批结果，更新 Aims 内状态

### 7.4 GitLab 集成策略

- 利用 Account 模块已有的 GitLab 项目同步能力获取仓库信息
- Aims 自身通过 GitLab API 拉取 commit 和 MR 数据
- 通过 commit message 约定（如 `#project_code-123`）实现自动关联
- 不做 WebHook 实时推送（MVP 阶段），采用定期同步 + 手动刷新

## 9. 风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| MVP 范围过大导致延期 | 高 | 严格控制 P0 范围，P1/P2 可延后 |
| Workflow 模块尚未完全稳定 | 中 | MVP 阶段先做简单的状态流转，审批功能可后接 |
| GitLab API 调用性能 | 中 | 做好缓存和增量同步 |
| 看板拖拽交互复杂度 | 中 | 采用成熟的拖拽库（如 vuedraggable 或 dnd-kit） |
| 与 Altoc 的工时数据接口 | 低 | Altoc 为远期规划，Aims 先做好自身工时记录 |
