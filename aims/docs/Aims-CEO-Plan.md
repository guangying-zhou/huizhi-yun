# CEO Plan: Aims MVP — 一体化研发项目管理

## Vision

### 10x Check

10x 版本的 Aims 不是"另一个项目管理工具"，而是**开发者的操作系统**——代码提交自动驱动任务状态、AI 自动拆分需求为可执行任务、代码审查结果自动流入质量报表。开发者几乎不需要主动"管理项目"，系统通过代码活动自动追踪进度。

与当前 MVP 的差距：MVP 仍然需要手动操作（创建任务、拖拽看板、录入工时），10x 版本的自动化程度远超当前。但 MVP 通过 GitLab 关联迈出了第一步。

### Platonic Ideal

完美版本中，项目经理打开 Aims 看到的不是任务列表，而是一个**实时的项目健康地图**：哪些需求在推进、哪些卡住了、哪些代码已合并但还没测试、哪些测试通过了但还没发布。所有信息自动从代码活动、文档变更、审批状态中聚合——无需任何人"更新状态"。

## Core Strategic Insight

**护城河在代码集成，不在一体化。** 飞书、钉钉、Tapd 都能做文档+审批+项目的打通。但没有人真正把代码提交→任务状态→代码审查→测试结果→发布版本这条链路融入中国企业的项目管理工具。GitLab 自己的 Issues 做到了，但缺少中国企业需要的审批流、组织架构、企业微信。这是汇智云的真正差异化。

**MVP 策略决策：** 尽管护城河在代码集成，创始人选择 MVP 先做全面的一体化体验（方案 B），在 Beta 阶段再深入代码集成（WebHook + MR + CI/CD）。理由：先用最小成本验证"一体化体验本身有价值"（假设 H1），再加大对护城河的投入。如果 H1 失败（用户不愿切换），则需重新评估——可能直接转向方案 D（代码优先）。

**MVP 验证的核心假设：**
- H1: 一体化体验（任务+文档+代码在同一界面）本身就足以让内部团队从禅道切换
- H2: 迭代看板 + 企业微信通知的组合比禅道的对应功能体验更好
- 验证标准见下方 Success Criteria

## Accepted Scope (added to this plan)

- **智能工作台**：首页展示"我的任务"（按优先级排序）、"今日动态"（关注项目的最新变更）、"我的工时"快捷录入、通知概览
- **批量操作**：列表视图支持多选 + 批量修改状态/优先级/指派人
- **键盘快捷键**：J/K 上下导航、X 选中、E 编辑、Enter 打开详情

## Deferred to TODOS.md

- **全局搜索 + Command Palette**：Cmd+K 快捷搜索任务/项目/页面。工作量 S。Beta 阶段实现。
- **自定义字段**：项目级自定义字段（文本/下拉/日期/数字），JSON Schema 存储 + 动态表单渲染。工作量 M。商业化阶段实现。
- **数据导入/导出**：CSV/Excel 导入导出任务和缺陷。工作量 S。内部推广前实现。
- **GitLab 深度集成**：WebHook 实时推送、MR 状态同步→任务状态自动流转、代码审查流程融入任务详情。工作量 M。Beta 阶段优先。

## Updated MVP Scope Summary

方案 B 原有功能 + 2 个新增扩展：

**P0 — 核心功能：**
- 项目 CRUD + 成员管理
- 项目业务状态机（draft/approval_pending/active/paused/completed/archived）
- 迭代/Sprint 管理
- 任务管理（看板 + 列表视图 + 批量操作 + 键盘快捷键）
- 缺陷跟踪
- 统一工作项模型
- 简单状态机（预留 Workflow 接口）

**P1 — 一体化联动 + 体验提升：**
- GitLab commit 基础关联（手动同步，基于 `project_code-序号`）
- Codocs 文档关联（Aims→Codocs 单向）
- 企业微信通知
- 基础工时记录
- 智能工作台（首页）

## Risk Assessment

| 风险 | 可能性 | 影响 | 应对 |
|------|--------|------|------|
| Workflow 不可用导致审批体验差 | 高 | 中 | 简单状态机替代，预留接口 |
| 内部团队不愿从禅道切换 | 中 | 高 | Assignment: 提前做用户访谈 |
| 代码集成"浅尝辄止"不够打动用户 | 中 | 中 | Beta 快速跟进深度集成 |
| MVP 范围扩大后延期 | 中 | 中 | 新增 2 个 S 级功能（+1 CC 天）；若延期超 1 周，砍掉键盘快捷键 |

## Success Criteria

| 指标 | 目标 | 度量方式 |
|------|------|---------|
| 内部团队采纳率 | 上线 2 周内 ≥ 50% 项目成员日活 | 每日登录且有操作 |
| 跨模块使用率 | ≥ 30% 任务关联了 Codocs 或 GitLab | 关联数据统计 |
| 禅道替代进度 | 上线 1 月后禅道日活下降 ≥ 50% | 对比数据 |
| 用户满意度 | NPS ≥ 30 | 上线后调研 |
