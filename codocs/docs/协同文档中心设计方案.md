# 协同文档中心设计方案

**适用模块：** `codocs`  
**版本：** 1.0  
**日期：** 2026-03-30

---

## 1. 背景

当前 `codocs` 的 `/mydocs/shared` 页面本质上仍是“共享文档页面”，只覆盖两类关系：

- 共享给我
- 我共享的

它依赖的是 `document_shares`，本质上是“个人共享视图”，不能覆盖以下真实协同场景：

- 个人原创文档创建后移交给部门，但原作者仍需继续查阅
- 对外发文中，发起人、审核人、监督人、盖章经办人都需要快速定位同一份文档
- 文档发布后，相关协同人员需要持续追溯发布版本
- 盖章部门此前可能并不属于原发文部门，按现有目录权限无法查看原文

因此，需要将“共享文档”升级为“协同文档中心”。

---

## 2. 目标

### 2.1 产品目标

将 `/mydocs/shared` 重构为“协同文档中心”，展示“我与文档的协同关系”，而不是“文档存放位置”。

### 2.2 业务目标

实现以下能力：

- 能快速定位所有与我有关的文档
- 能基于协同关系查看全文
- 不复制文档实体，只保存链接关系
- 发布后的文档统一只读
- 覆盖共享、原创移交、对外发文三类核心协同场景

---

## 3. 设计原则

### 3.1 单一实体

协同文档中心只展示链接，不创建文档副本。

- 文档仍然只有一份真实记录，存放在原有 `documents` 表和 OSS 路径中
- 协同中心仅建立“关系索引层”

### 3.2 关系优先

协同中心展示逻辑基于“我为什么能看到它”，而不是“它存在哪个目录”。

### 3.3 全文可见

只要用户与文档存在有效协同关系，即可查看全文。

适用对象包括但不限于：

- 共享接收人
- 原创作者
- 移交后原作者
- 对外发文发起人
- 审核人
- 业务部门审核人
- 委员会成员
- 分管领导
- 上级领导
- 监督人
- 总经办盖章经办人

### 3.4 编辑权分离

全文可见不等于可编辑。

- 未发布文档：编辑权限仍按原有权限模型判断
- 已发布文档：一律只读，不允许编辑

### 3.5 发布版本优先

对已发布文档，协同中心应优先定位到发布版本，而不是过程草稿。

---

## 4. 信息架构

建议将现有“共享文档”页面升级为“协同文档”页面，主入口仍可沿用 `/mydocs/shared`，后续视情况改为 `/mydocs/collaboration`。

### 4.1 一级分组

- 共享文档
- 原创文档
- 对外发文

### 4.2 二级筛选

每个一级分组下统一支持以下筛选：

- 待我处理
- 我发起的
- 我参与的
- 已完成

### 4.3 辅助筛选

- 按发起人筛选
- 按部门筛选
- 按状态筛选
- 按时间筛选
- 关键词搜索

### 4.4 列表展示字段

建议统一展示：

- 文档标题
- 当前归属位置
- 协同关系
- 当前状态
- 发起人/共享人
- 所属部门
- 更新时间
- 是否已发布

### 4.5 详情侧栏

点击后侧栏或右侧预览区展示：

- 全文预览
- 文档元信息
- 协同关系说明
- 当前流程状态
- 发布记录

## 5. 关系模型

协同文档中心不再直接耦合 `document_shares` 的查询形态，而是引入统一的“文档关系索引”。

建议新增表：`document_relations`

### 5.1 表用途

记录“某个用户为什么能在协同中心看到某个文档”。

### 5.2 推荐字段

```sql
CREATE TABLE document_relations (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  document_id BIGINT UNSIGNED NOT NULL,
  document_uuid CHAR(36) NOT NULL,
  related_uid VARCHAR(64) NOT NULL,
  relation_type VARCHAR(50) NOT NULL,
  source_type VARCHAR(50) NOT NULL,
  source_id VARCHAR(100) DEFAULT NULL,
  source_dept_code VARCHAR(50) DEFAULT NULL,
  can_read TINYINT NOT NULL DEFAULT 1,
  can_edit TINYINT NOT NULL DEFAULT 0,
  can_comment TINYINT NOT NULL DEFAULT 0,
  status TINYINT NOT NULL DEFAULT 1,
  metadata JSON DEFAULT NULL,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  KEY idx_related_uid (related_uid, status, updated_at),
  KEY idx_document_uuid (document_uuid, status),
  UNIQUE KEY uk_relation (document_id, related_uid, relation_type, source_type, source_id)
)
```

### 5.3 relation_type 建议值

- `shared_to_me`
- `shared_by_me`
- `created_by_me`
- `transferred_by_me`
- `outside_initiator`
- `outside_reviewer`
- `outside_supervisor`
- `outside_seal_handler`

### 5.4 source_type 建议值

- `document_share`
- `document_transfer`
- `review`
- `manual`

---

## 6. 权限矩阵

### 6.1 基本原则

- 只要存在有效协同关系，默认 `can_read = 1`
- 发布后文档一律 `can_edit = 0`
- 编辑权限不由协同中心提升，只能继承原始权限

### 6.2 未发布文档

#### 共享文档

- 被共享人：可读；是否可写取决于共享权限
- 共享人：可读可写

#### 原创移交文档

- 原作者：可读
- 当前归属部门：按部门文档权限控制编辑

#### 对外发文流程中

- 发起人：可读；是否可编辑取决于当前流程阶段和原文权限
- 当前审阅人：可读全文
- 其他已参与节点人员：可读全文
- 监督/盖章人员：可读全文

### 6.3 已发布文档

统一规则：

- 只读
- 可查看全文
- 不允许编辑
- 如需变更，必须从原始文档修订并重新发起流程

### 6.4 特殊说明

对外发文的盖章经办人需要核对打印件与原文，因此必须可查看全文，但不应获得编辑权。

## 7. 三类协同场景设计

### 7.1 共享文档

来源：

- `document_shares`

展示规则：

- “共享给我”对应 `shared_to_me`
- “我共享的”对应 `shared_by_me`

说明：

- 这部分可沿用现有能力，只是接入统一关系模型

### 7.2 原创文档

定义：

- 用户个人创建的文档，后续正式移交给部门继续维护

目标：

- 文档移交后，原作者仍可在协同中心快速找到该文档
- 不改变文档实际归属目录

建议做法：

- 新增“移交”行为
- 记录原作者与文档的持续关系 `created_by_me` / `transferred_by_me`

说明：

当前系统还没有完整的“原创移交”业务模型，因此这部分应在第二阶段建设。

### 7.3 对外发文

来源：

- `document_reviews`
- `review_actions`
- `flow_snapshot`
- `extra`

应写入协同关系的人：

- 发起人
- 部门经理
- 业务部门经理
- 委员会成员
- 分管领导
- 上级领导
- 总经办盖章经办人

说明：

- 对外发文在发布后，协同中心应优先指向发布版本
- 如未发布，则指向审阅中的原文

---

## 8. 数据写入策略

### 8.1 实时写入

在以下事件发生时同步维护 `document_relations`：

- 文档共享
- 撤销共享
- 文档移交
- 提交审阅
- 审批节点推进
- 确认发布

### 8.2 对外发文关系写入时机

#### 提交审阅时

根据 `flow_snapshot` 解析出全部参与人，写入：

- `outside_initiator`
- `outside_reviewer`
- `outside_supervisor`

如勾选“需要加盖公章”，同时为总经办成员写入：

- `outside_seal_handler`

#### 确认发布时

- 将协同关系继续保留
- 将关联目标切换为发布版本文档 UUID
- 原过程文档关系可保留历史标识，或标记为已归档来源

## 9. API 设计

### 9.1 协同文档列表

```http
GET /api/collab-docs
```

建议参数：

- `category=shared|original|outside`
- `scope=todo|initiated|participated|done`
- `keyword`
- `dept_code`
- `owner_uid`
- `status`
- `page`
- `limit`

返回字段建议：

- `document_uuid`
- `title`
- `doc_type`
- `dept_code`
- `owner_uid`
- `relation_type`
- `source_type`
- `status`
- `readonly_flag`
- `publish_info`
- `updated_at`

### 9.2 协同文档详情

```http
GET /api/collab-docs/:uuid
```

返回内容：

- 文档基础信息
- 当前协同关系
- 全文内容
- 流程信息
- 发布信息

### 9.3 协同文档关系重建

```http
POST /api/collab-docs/rebuild
```

用途：

- 历史数据补建
- 修复关系索引不一致

## 10. 实现路径

### 10.1 第一阶段

目标：先把“共享文档”升级为“协同文档中心 MVP”

范围：

- 页面改版
- 新增 `document_relations`
- 接入共享文档关系
- 接入对外发文关系
- 已发布文档只读

不做：

- 原创文档移交能力
- 复杂权限分层

### 10.2 第二阶段

目标：补齐原创移交和更细化的工作流聚合

范围：

- 新增文档移交模型
- 接入原创文档
- 增加“待我处理 / 我参与 / 已完成”统计
- 增加关系修复与后台管理工具

---

## 11. 对现有系统的影响

### 11.1 前端

需重构：

- `/app/pages/mydocs/shared.vue`

建议调整：

- 页面名称从“共享文档”改为“协同文档”
- Tab 逻辑从“双 tab”改为“一级分类 + 二级筛选”
- 引入统一文档预览与关系标签

### 11.2 后端

需新增：

- `document_relations` 表
- `/api/collab-docs` 系列接口

需改造：

- 文档共享接口写入关系索引
- 对外发文提交与发布流程写入关系索引
- `requireDocumentReadAccess` 增加“协同关系可读”判断

### 11.3 权限

现有 `requireDocumentReadAccess()` 只认：

- 所有者
- 直接共享
- 部门文档部门访问

需要扩展为：

- 若用户命中有效 `document_relations.can_read = 1`，则允许查看全文

但仍保持：

- 写权限按原始权限模型判断
- 已发布文档统一只读

---

## 12. 关键决策

本方案确定以下原则：

1. 协同文档中心展示的是“关系”，不是“副本”
2. 关系一旦成立，即允许查看全文
3. 发布后的文档统一只读
4. 编辑权限不因协同关系自动提升
5. 对外发文发布后优先展示发布版本
6. 总经办盖章人员属于合法协同关系人，必须可见全文

---

## 13. 风险与注意事项

### 13.1 历史数据补建

已有共享记录、已有对外发文流程都需要回填 `document_relations`。

### 13.2 关系去重

同一用户可能同时命中多个关系：

- 发起人
- 审核人
- 盖章经办

页面展示应支持多标签，但索引层要避免重复脏数据。

### 13.3 发布前后指向切换

对外发文从“审阅过程文档”切换到“发布版本文档”时，关系索引必须同步切换，否则协同中心会出现定位错误。

### 13.4 可见与可编辑分离

这是整个方案最重要的边界，不能被打穿。

---

## 14. 结论

“共享文档”升级为“协同文档中心”是合理且必要的。

从系统设计上，应将它定义为一层统一的“文档关系索引层”，而不是继续沿用 `document_shares` 做页面拼接。

推荐实施策略：

- 第一阶段先接“共享文档 + 对外发文”
- 第二阶段再补“原创移交”
- 在底层建立统一的 `document_relations`
- 在访问控制中引入“协同关系可读”
- 坚持“发布后只读、全文可见、无副本”

这会比继续维护“共享文档页面”更稳定，也更符合你们当前文档协作和发文管理的真实业务形态。
