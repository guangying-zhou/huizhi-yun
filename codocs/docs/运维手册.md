# Codocs 运维手册

本手册面向系统管理员，涵盖 Codocs 模块的日常运维操作。

Base URL 示例：`http://localhost:3001`（生产环境请替换为实际地址）

---

## 1. OSS 存储清理

Codocs 使用阿里云 OSS 存储文档和图片。随着使用积累，OSS 上会产生孤立文件（数据库中已无对应记录），需要定期清理。

所有清理接口默认为**预览模式（dryRun）**，不会执行删除。确认无误后再传参执行。

### 1.1 清理历史遗留 YJS 文件

当前协同编辑会将 `.yjs` 文件作为 Yjs CRDT 快照持久化到 OSS，用于避免重连/重启后协同状态重建导致的内容重复。

`DELETE /api/admin/cleanup-yjs` 仅适用于清理**历史遗留、明确已废弃路径**下的 `.yjs` 文件，不应对当前仍在使用的文档路径批量执行。

**接口：** `DELETE /api/admin/cleanup-yjs`

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `dryRun` | string | `'true'` | `'true'` 仅预览，`'false'` 执行删除 |
| `prefix` | string | 空 | 指定扫描前缀，默认扫描 `codocs/` 和 `recycle.bin/` |

**预览：**

```bash
curl -X DELETE 'http://localhost:3001/api/admin/cleanup-yjs?dryRun=true'
```

**执行删除：**

```bash
curl -X DELETE 'http://localhost:3001/api/admin/cleanup-yjs'
```

> 注：该接口只应用于一次性的历史清理。对现网活跃文档执行删除，会导致协同状态丢失并在后续重连时重新走 Markdown 回退流程。

**指定前缀扫描：**

```bash
curl -X DELETE 'http://localhost:3001/api/admin/cleanup-yjs?dryRun=true&prefix=codocs/users/'
```

**返回示例：**

```json
{
  "success": true,
  "dryRun": true,
  "mainBucket": { "count": 42, "files": [...] },
  "projectsBucket": { "count": 3, "files": [...] },
  "totalCount": 45,
  "totalSizeMB": 12.35
}
```

### 1.2 清理孤立文档文件

扫描 OSS 上的 `.md` 文件，与数据库 `documents` 表比对，找出无对应记录的孤立文件。

**接口：** `DELETE /api/admin/cleanup-orphan-docs`

**参数：**

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `dryRun` | string | `'true'` | `'true'` 仅预览，`'false'` 执行删除 |
| `prefix` | string | 空 | 指定扫描前缀 |
| `includeRecycleBin` | string | `'false'` | `'true'` 同时扫描回收站 |

**预览（默认不删除）：**

```bash
curl -X DELETE 'http://localhost:3001/api/admin/cleanup-orphan-docs'
```

**执行删除：**

```bash
curl -X DELETE 'http://localhost:3001/api/admin/cleanup-orphan-docs?dryRun=false'
```

**包含回收站残留：**

```bash
curl -X DELETE 'http://localhost:3001/api/admin/cleanup-orphan-docs?includeRecycleBin=true'
```

**判定规则：**

| OSS 位置 | 数据库状态 | 判定 |
|----------|-----------|------|
| `codocs/...` | 有活跃记录 (status!=0) | 保留 |
| `codocs/...` | 有已删除记录 (status=0) | 保留 |
| `codocs/...` | 无记录 | 孤立，可删除 |
| `recycle.bin/...` | 有已删除记录指向它 | 保留（可恢复） |
| `recycle.bin/...` | 无记录 | 孤立，可删除 |

> `info/` 目录下的文件（前沿资讯/推荐文章）由 x-bookmark-fetcher 管理，不在此清理范围内。

**返回示例：**

```json
{
  "success": true,
  "dryRun": true,
  "summary": {
    "ossFilesScanned": 320,
    "dbDocuments": 280,
    "activeDocuments": 250
  },
  "orphanFiles": {
    "mainBucket": [...],
    "recycleBin": [...],
    "projectsBucket": [...]
  },
  "totalOrphans": 15,
  "totalSizeMB": 2.35
}
```

### 1.3 清理孤立图片

扫描 Images Bucket 中的图片文件，检测关联文档是否存在及是否仍被引用。

**扫描图片：** `GET /api/admin/images`

```bash
curl 'http://localhost:3001/api/admin/images'
```

返回所有图片及其状态：

| orphan 状态 | 含义 |
|-------------|------|
| 空字符串 | 正常（文档存在且引用中） |
| `no-meta` | 无元数据的旧图片（无法判断归属） |
| `deleted` | 关联文档已删除 |
| `unreferenced` | 文档存在但不再引用该图片 |

**删除孤立图片：** `DELETE /api/admin/images`

```bash
curl -X DELETE 'http://localhost:3001/api/admin/images' \
  -H 'Content-Type: application/json' \
  -d '{"paths": ["codocs/users/xxx/images/abc.png", "codocs/users/xxx/images/def.jpg"]}'
```

> 安全限制：只允许删除孤立图片（文档已删除/未被引用/无元数据），正在被引用的图片会被拒绝删除。

## 2. OSS 存储结构

### 2.1 Bucket 分布

| Bucket | 用途 | 访问权限 |
|--------|------|----------|
| Main Bucket | 文档 `.md` 文件、文件柜附件 | 私有（签名 URL） |
| Projects Bucket | Git 项目文档（`doc_type='git-project'`） | 私有 |
| Images Bucket | 所有用户上传的图片 | 公共读 |

### 2.2 路径规则

**文档：**

| 文档类型 | OSS 路径 |
|----------|----------|
| 个人文档 | `codocs/users/{uid}/docs/{folder}/{title}.md` |
| 幻灯片 | `codocs/users/{uid}/slides/{title}.md` |
| 部门文档 | `codocs/departments/{dept_code}/docs/{title}.md` |
| 项目文档 | `codocs/projects/{project_code}/docs/{title}.md` |
| 销售文档 | `codocs/sale/{entity_code}/docs/{title}.md` |
| 公司发文 | `codocs/publish/company/{title}.md` |
| 知识库 | `codocs/publish/knowledge/{title}.md` |
| 产品文档 | `codocs/publish/products/{title}.md` |
| 工作日志 | `codocs/worklogs/{uid}/{date}-{name}工作日志.md` |
| 工作周报 | `codocs/worklogs/{uid}/weekly/{year}-W{week}-{name}工作周报.md` |
| 回收站 | `recycle.bin/...`（与原路径结构相同） |

**图片：**

| 来源 | OSS 路径 | Bucket |
|------|----------|--------|
| 编辑器上传 | `codocs/users/{uid}/images/{uuid}.{ext}` | Images |
| Issue 图片 | `codocs/issues/images/{uuid}.{ext}` | Images |
| X 书签文章 | `codocs/info/images/{bookmark_id}_{idx}_{hash}.{ext}` | Main |

**文件柜：**

| 类型 | OSS 路径 |
|------|----------|
| 个人文件柜 | `codocs/users/{uid}/cabinet/{uuid}.{ext}` |
| 部门文件柜 | `codocs/departments/{dept_code}/cabinet/{uuid}.{ext}` |

## 3. 服务管理

### 3.1 Codocs 主服务（Nuxt/Nitro）

```bash
cd codocs/
pnpm dev                    # 开发模式（端口 3001）
pnpm build                  # 生产构建（需增加内存）
NODE_OPTIONS='--max-old-space-size=4096' nuxt build --dotenv .env
```

### 3.2 Collab Runtime 协作服务

```bash
cd ../collab
pnpm dev                    # standalone 开发模式（端口 3021）
```

生产默认由 Console embedded 启动 Collab Runtime；需要独立扩容或故障隔离时再按 `collab/README.md` 使用 standalone 模式部署。

协作状态会同时落盘两份：

- `.yjs`：Yjs CRDT 二进制快照，作为协同状态的真实恢复来源
- `.md`：Markdown 文本快照，供预览、导出、版本记录和非协同读取使用

服务启动或文档重载时优先恢复 `.yjs`，仅在历史文档尚无 `.yjs` 快照时才回退到 `.md`。

### 3.3 X Bookmark Fetcher

```bash
cd codocs/x-bookmark-fetcher/
python -m app.main           # 启动服务（端口 8001）
```

## 4. 数据库维护

### 4.1 Schema 文件

数据库表结构定义在 `codocs/docs/codocs_schema.sql`，数据库名 `hzy_codocs`。

### 4.2 文档软删除

删除文档时执行软删除：`status` 设为 0，`deleted_at` 记录时间，OSS 文件移到 `recycle.bin/` 前缀。

恢复文档时从回收站移回原路径，`status` 恢复为 1。

### 4.3 建议的定期维护

| 任务 | 频率 | 操作 |
|------|------|------|
| 清理孤立文档 | 每月 | `DELETE /api/admin/cleanup-orphan-docs` |
| 清理孤立图片 | 每月 | `GET /api/admin/images` + `DELETE /api/admin/images` |
| 清理 YJS 文件 | 一次性 | `DELETE /api/admin/cleanup-yjs`（仅清理确认废弃的历史遗留文件） |
