# 对外发文发布后执行方案

**适用模块：** `account`、`codocs`  
**适用范围：** 对外发文审批通过后的发布、盖章、发送与留痕  
**版本：** 1.1  
**日期：** 2026-03-30

---

## 1. 目标

在现有“审批通过 -> 发起人确认发布”基础上，补齐对外发文发布后的执行闭环：

- 需要盖章的文件，增加“确认盖章”
- 已盖章或无需盖章的文件，增加“确认发送”
- 系统形成完整执行留痕，支持后续追溯

本方案明确区分：

- 审批状态：文件是否完成审核
- 执行状态：文件发布后是否完成盖章、发送

执行状态不并入现有 `document_reviews.status`，避免与审批流混淆。

---

## 2. 总体原则

### 2.1 审批与执行分离

现有审批流保持不变：

1. 提交审阅
2. 审批流推进
3. 审批通过
4. 发起人确认发布

新增发布后执行环节：

5. 若需要盖章，进入“待盖章”
6. 公章管理员确认盖章
7. 进入“待发送”
8. 业务人员确认发送
9. 进入“已发送”

### 2.2 不增加文档副本

- 文档仍只保留发布版本一份
- 盖章、发送只形成记录，不产生新的系统副本
- 协同文档中心与部门文档页均定位到同一份已发布文档

### 2.3 发布后统一只读

- 已发布文档统一只读
- 盖章经办、发送经办仅能查看全文与登记记录
- 不获得正文编辑权

---

## 3. 状态设计

### 3.1 现有状态

现有 `document_reviews.status` 保持：

- `pending`
- `in_progress`
- `approved`
- `rejected`
- `archived`

其中：

- `approved` = 审批通过，待发起人确认发布
- `archived` = 已确认发布，已生成发布版本

### 3.2 新增执行状态

建议在 `document_reviews` 增加字段：

- `execution_status VARCHAR(30) DEFAULT NULL`

建议枚举：

- `NULL`：非对外发文或尚未确认发布
- `pending_seal`：待盖章
- `pending_send`：待发送
- `sent`：已发送

推荐规则：

- 对外发文确认发布后：
  - `needsOfficialSeal = true` -> `execution_status = 'pending_seal'`
  - `needsOfficialSeal = false` -> `execution_status = 'pending_send'`
- 确认盖章后：
  - `execution_status = 'pending_send'`
- 确认发送后：
  - `execution_status = 'sent'`

说明：

- `sealed` 可以不单独暴露为长期状态；实现上也可以直接从 `pending_seal -> pending_send`
- 若希望保留更细时间点，可记录 `sealed_at`，不一定需要单独状态

## 4. 数据表设计

### 4.1 盖章记录表

建议新增：

```sql
CREATE TABLE document_seal_records (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  review_id BIGINT NOT NULL COMMENT '关联 document_reviews.id',
  document_uuid VARCHAR(36) NOT NULL COMMENT '发布版本文档 UUID',
  seal_types JSON NOT NULL COMMENT '盖章类型数组：公章/法人章/财务章/合同章',
  page_count INT NOT NULL COMMENT '文档页数',
  operator_uid VARCHAR(64) NOT NULL COMMENT '盖章确认人',
  confirmed_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '确认盖章时间',
  remark VARCHAR(500) DEFAULT NULL COMMENT '备注',
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  INDEX idx_review_id (review_id),
  INDEX idx_document_uuid (document_uuid),
  INDEX idx_operator_uid (operator_uid)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

约束建议：

- 同一 `review_id` 可允许多条记录，适配补章或追加章型
- 若业务要求“正常情况下只确认一次”，则前端默认只允许首次登记，追加登记留给后续扩展

### 4.2 发送记录表

建议新增：

```sql
CREATE TABLE document_send_records (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  review_id BIGINT NOT NULL COMMENT '关联 document_reviews.id',
  document_uuid VARCHAR(36) NOT NULL COMMENT '发布版本文档 UUID',
  sender_uid VARCHAR(64) NOT NULL COMMENT '发送人',
  receiver_name VARCHAR(100) NOT NULL COMMENT '接收人',
  receiver_phone VARCHAR(30) NOT NULL COMMENT '联系电话',
  channel VARCHAR(20) NOT NULL COMMENT '发送途径：email/wecom/wechat/qq/usb',
  target_account VARCHAR(200) DEFAULT NULL COMMENT '对方账号，U盘时为空',
  confirmed_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '确认发送时间',
  remark VARCHAR(500) DEFAULT NULL COMMENT '备注',
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  INDEX idx_review_id (review_id),
  INDEX idx_document_uuid (document_uuid),
  INDEX idx_sender_uid (sender_uid)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

约束建议：

- `channel = 'usb'` 时允许 `target_account IS NULL`
- 其他途径要求 `target_account` 必填
- 同一 `review_id` 默认允许多条发送记录，适配一文多次发送

### 4.3 审阅表扩展

建议新增字段：

```sql
ALTER TABLE document_reviews
  ADD COLUMN execution_status VARCHAR(30) DEFAULT NULL COMMENT '发布后执行状态',
  ADD COLUMN published_document_uuid VARCHAR(36) DEFAULT NULL COMMENT '发布版本文档 UUID',
  ADD COLUMN sealed_at DATETIME DEFAULT NULL COMMENT '确认盖章时间',
  ADD COLUMN sent_at DATETIME DEFAULT NULL COMMENT '确认发送时间';
```

说明：

- `published_document_uuid` 便于不再通过 `archive_oss_path` 反查
- `sealed_at`、`sent_at` 便于列表筛选与统计

## 5. 公章管理员设计

### 5.1 结论

不建议直接在 `system_users` 表增加 `is_seal_admin` 字段。  
建议优先采用 `account` 现有 RBAC 模型，通过角色控制。

### 5.2 原因

- “公章管理员”是职责，不是用户固有属性
- 后续可能出现不同章型由不同人管理
- 后续可能按公司、组织范围区分
- 现有 `account` 已有 `roles`、`user_roles`，复用成本最低

### 5.3 推荐落地

第一阶段最小方案：

- 新增角色：`seal_admin`
- 由管理员在 `account` 给用户赋该角色
- 该角色可执行“确认盖章”

第二阶段扩展方案：

如后续需按章型区分，再新增配置表：

```sql
CREATE TABLE seal_admin_scopes (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  uid VARCHAR(64) NOT NULL,
  seal_type VARCHAR(20) NOT NULL COMMENT 'official/legal/finance/contract',
  company_code VARCHAR(7) DEFAULT NULL,
  status TINYINT NOT NULL DEFAULT 1,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uk_uid_seal_type_company (uid, seal_type, company_code)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

当前阶段建议先不上 `seal_admin_scopes`，避免过度设计。

## 6. 页面与入口设计

### 6.1 协同文档中心：确认盖章

入口位置：

- `个人文档 / 协同文档`
- 仅对满足以下条件的用户显示：
  - 文档为对外发文
  - `execution_status = 'pending_seal'`
  - 当前用户拥有 `seal_admin` 角色

按钮位置：

- 文档右上角显示 `确认盖章`

弹窗字段：

- `盖章类型`：多选
  - 公章
  - 法人章
  - 财务章
  - 合同章
- `文档页数`：必填，正整数
- `备注`：可选

提交后动作：

1. 新增 `document_seal_records`
2. 更新 `document_reviews.execution_status = 'pending_send'`
3. 更新 `sealed_at`
4. 发送企业微信通知给发起人
5. 协同文档中心状态更新为“待发送”

### 6.2 部门文档 / 对外发文：确认发送

入口位置：

- `部门文档 / 对外发文`
- 仅对满足以下条件的用户显示：
  - 文档为对外发文发布版本
  - `execution_status = 'pending_send'`

按钮位置：

- 文档操作区显示 `确认发送`

弹窗字段：

- `发送人`：必填，下拉选择公司用户
- `接收人`：必填
- `联系电话`：必填
- `发送途径`：必填，单选
  - 邮件
  - 企业微信
  - 微信
  - QQ
  - U盘
- `对方账号`：除 `U盘` 外必填
- `备注`：可选

提交后动作：

1. 新增 `document_send_records`
2. 更新 `document_reviews.execution_status = 'sent'`
3. 更新 `sent_at`
4. 发送企业微信通知给发起人
5. 页面状态更新为“已发送”

## 7. API 设计

### 7.1 确认发布接口调整

现有：

- `POST /api/reviews/:id/archive`

需要补充：

- 对外发文确认发布后，写入 `execution_status`
- 写入 `published_document_uuid`

推荐规则：

- `needsOfficialSeal = true` -> `pending_seal`
- 否则 -> `pending_send`

### 7.2 确认盖章接口

建议新增：

- `POST /api/reviews/:id/seal`

请求体：

```json
{
  "sealTypes": ["official", "contract"],
  "pageCount": 3,
  "remark": "纸质件核对无误"
}
```

校验规则：

- 当前用户必须具备 `seal_admin`
- 当前审阅必须是 `对外发文`
- 当前状态必须为 `archived`
- `execution_status` 必须为 `pending_seal`
- `sealTypes` 至少选择一个
- `pageCount >= 1`

返回：

- 当前执行状态
- 新增盖章记录 ID

### 7.3 确认发送接口

建议新增：

- `POST /api/reviews/:id/send`

请求体：

```json
{
  "senderUid": "zhangsan",
  "receiverName": "李四",
  "receiverPhone": "13800000000",
  "channel": "email",
  "targetAccount": "lisi@example.com",
  "remark": "已发送正式版"
}
```

校验规则：

- 当前审阅必须是 `对外发文`
- 当前状态必须为 `archived`
- `execution_status` 必须为 `pending_send`
- `channel != usb` 时，`targetAccount` 必填

返回：

- 当前执行状态
- 新增发送记录 ID

### 7.4 记录查询方式

当前实现未单独拆分：

- `GET /api/reviews/:id/seal-records`
- `GET /api/reviews/:id/send-records`

而是通过以下接口直接返回相关记录：

- `GET /api/reviews/:id`
- `GET /api/reviews/by-oss-path`

用途：

- 审阅详情展示
- 发布记录弹窗查看
- 部门文档 / 对外发文查看

## 8. 权限设计

### 8.1 盖章权限

推荐新增资源动作：

- 资源：`codocs_reviews`
- 动作：`seal_confirm`

赋予角色：

- `seal_admin`

前后端同时校验：

- 前端决定按钮是否显示
- 后端再次校验，避免绕过页面直接调用接口

### 8.2 发送权限

当前实现口径：

- 发起人可确认发送
- 发文部门具备部门文档写权限的经办人可确认发送

说明：

- 这里复用了现有部门文档写权限模型
- 这样既允许发起人自行登记，也允许实际代发人员完成留痕

---

## 9. 通知设计

### 9.1 确认发布后

- 若需盖章：通知具备 `seal_admin` 角色的公章管理员，状态为“待盖章”
- 若无需盖章：仅更新状态为“待发送”，不额外发送专门通知

### 9.2 确认盖章后

- 通知发起人，状态变更为“待发送”

### 9.3 确认发送后

- 通知发起人
- 可选通知监督人/上级领导

---

## 10. 列表与展示规则

### 10.1 协同文档中心

对外发文文档建议新增展示字段：

- 审批状态
- 执行状态
- 是否需盖章
- 是否已盖章
- 是否已发送

按钮显示规则：

- `pending_seal + seal_admin` -> 显示 `确认盖章`
- 其他状态不显示

### 10.2 部门文档 / 对外发文

文档卡片或详情建议显示：

- 文件级别
- 是否需盖章
- 执行状态
- 最近一次发送记录摘要

按钮显示规则：

- `pending_send` -> 显示 `确认发送`

当前实现：

- “确认发送”按钮显示在右上角文档操作区
- 发布记录弹窗可查看完整盖章记录与发送记录

## 11. 审阅详情页

当前实现已在 `reviews/[id].vue` 增加执行信息展示，包含：

- 发布状态：已发布

- 盖章记录
- 发送记录

这样审阅详情页可以作为完整的流程留痕页。

## 13. 推荐结论

当前推荐结论：

1. 审批状态与执行状态已分离
2. 公章管理员当前采用 `account` 角色 `seal_admin`
3. 发布后已实现“确认盖章”与“确认发送”
4. 所有动作只生成记录，不复制文档
5. 已发布文档继续保持统一只读规则

这样实现成本可控，且与现有 `account` RBAC、`codocs` 发布归档、协同文档中心三部分结构一致。
