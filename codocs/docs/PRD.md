## 1. 产品概述

### 1.1 产品定位

Codocs 是 huizhi-yun 项目中的核心文档管理模块，致力于为企业提供统一的文档管理平台，并为全体员工提供高效的文档协同编辑服务。本模块依赖Account模块，通过Account模块的API接口（见Account-API.md）获取部门、用户、项目等信息，与GitLab进行交互，实现文档的同步、冲突解决、提交等功能。

### 1.2 核心价值

| 维度         | 价值描述                                             |
| ------------ | ---------------------------------------------------- |
| **统一管理** | 文档集中存储于阿里云 OSS，实现企业知识资产的统一归集 |
| **实时协同** | 基于 Milkdown 的多人在线协同编辑，解决团队协作痛点   |
| **AI 增强**  | AI 辅助编辑能力，提升全员文档编写效率                |
| **全员友好** | 兼顾技术人员和非技术人员的使用习惯                   |

- **服务端**：通过 Collab Runtime 提供协作

### 4.7 项目文档管理

#### 4.7.1 功能概述

项目文档管理功能为用户提供便捷的项目文档查看、同步和提交能力，实现项目文档与 GitLab 仓库的关联管理。

#### 4.7.2 界面布局

```mermaid
graph LR
    subgraph "项目文档管理页面"
        Left[左侧：项目列表] --> Right[右侧：文档列表+操作区]
        Right --> Sync[同步按钮]
        Right --> Commit[提交按钮]
    end
```

#### 4.7.3 功能流程

```mermaid
sequenceDiagram
    participant User as 用户
    participant Pinia as Pinia Store
    participant API as Account API
    participant OSS as 阿里云 OSS
    participant DB as Documents 表

    Note over User,DB: 1. 加载项目列表
    User->>Pinia: 访问项目文档管理页面
    Pinia->>API: 调用 Account 模块接口
    API-->>Pinia: 返回当前用户的项目列表
    Pinia-->>User: 在页面左侧展示项目列表

    Note over User,DB: 2. 点击项目查看文档
    User->>Pinia: 点击某个项目
    Pinia->>Pinia: 校验 repo_url

    alt repo_url 有值 && 以 GITLAB_BASE_URL 开头 && docs_synced_at 有值
        Pinia->>OSS: 获取 [repo_url - GITLAB_BASE_URL] 路径的文件列表
        Note over OSS: 包含子目录
        OSS-->>Pinia: 返回文件列表
        Pinia->>DB: 保存文件列表至 documents 表
        Pinia-->>User: 在右侧展示文档列表
    else repo_url 不符合条件
        Pinia-->>User: 提示：未配置项目仓库地址或尚未进行项目文档同步
    end

    Note over User,DB: 3. 项目文档同步
    User->>Pinia: 点击"同步"按钮
    Pinia->>API: 调用 API 4.7（项目文档同步）
    API->>OSS: 从 GitLab 同步文档至 OSS
    API-->>Pinia: 同步完成，返回新的 docs_synced_at
    Pinia->>OSS: 重新获取文件列表
    OSS-->>Pinia: 返回最新文件列表
    Pinia->>DB: 更新 documents 表
    Pinia-->>User: 刷新文档列表展示

    Note over User,DB: 4. 项目文档提交
    User->>Pinia: 点击"提交"按钮
    Pinia->>Pinia: 比较 docs_committed_at 与文档最后修改时间
    Pinia->>Pinia: 筛选需要提交的文件
    Pinia->>API: 调用 API 4.8（项目文档提交）
    Note over API: 提交参数：需要提交的文件列表
    API->>OSS: 将修改的文档提交至 GitLab
    API-->>Pinia: 提交完成，返回新的 docs_committed_at
    Pinia-->>User: 提示提交成功
```

#### 4.7.4 详细需求

| 功能点             | 详细说明                                                                                                                                                                                        |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **项目列表展示**   | 1. 通过 Pinia 调用 Account 模块接口获取当前用户的项目列表<br>2. 在页面左侧以列表形式展示所有项目<br>3. 显示项目名称、同步状态等基本信息                                                         |
| **文档列表查看**   | 1. 点击项目时，先校验 `repo_url` 和 `docs_synced_at` 字段<br>2. 如果校验通过，从 OSS 获取 `repo_url - GITLAB_BASE_URL` 路径的文件列表（包含子目录）<br>3. 将文件列表保存至 `documents` 表并展示 |
| **条件校验**       | 项目需要同时满足以下条件才能查看文档：<br>1. `repo_url` 有值<br>2. `repo_url` 以 `GITLAB_BASE_URL` 开头<br>3. `docs_synced_at` 有值（已进行过同步）                                             |
| **未配置项目提示** | 如果项目不满足查看条件，展示提示信息：<br>"未配置项目仓库地址或尚未进行项目文档同步"                                                                                                            |
| **文档同步功能**   | 1. 页面提供"同步"按钮<br>2. 点击后调用 API 4.7 接口进行项目文档同步<br>3. 同步完成后，更新 `docs_synced_at` 字段<br>4. 自动刷新并展示最新的文档列表                                             |
| **文档提交功能**   | 1. 页面提供"提交"按钮<br>2. 根据项目的 `docs_committed_at` 与文档最后修改时间比较<br>3. 筛选出需要提交的文件<br>4. 调用 API 4.8 接口提交文档<br>5. 提交成功后更新 `docs_committed_at` 字段      |
| **文件列表展示**   | 1. 支持树形结构展示文件和子目录<br>2. 显示文件名、修改时间、文件大小等信息<br>3. 标识哪些文件已修改需要提交                                                                                     |
| **数据存储**       | 文件列表信息保存至 `documents` 表，包括：<br>- 文件路径<br>- 文件名<br>- 最后修改时间<br>- 文件大小<br>- 关联的项目 ID                                                                          |

#### 4.7.5 技术要点

| 技术点           | 说明                                                  |
| ---------------- | ----------------------------------------------------- |
| **状态管理**     | 使用 Pinia 管理项目列表、文档列表、同步状态等全局状态 |
| **Account 集成** | 调用 Account 模块接口获取用户的项目列表               |
| **OSS 文件操作** | 使用阿里云 OSS SDK 获取指定路径的文件列表（含子目录） |
| **数据库操作**   | 将文件列表保存至 `documents` 表，支持批量插入和更新   |
| **文件变更检测** | 通过比较 `docs_committed_at` 和文件修改时间判断变更   |
| **API 4.7 调用** | 项目文档同步接口，从 GitLab 同步文档至 OSS            |
| **API 4.8 调用** | 项目文档提交接口，将修改的文档提交至 GitLab           |
| **环境变量**     | `GITLAB_BASE_URL` 作为环境变量配置，用于 URL 匹配判断 |

#### 4.7.6 用户权限

| 操作         | 权限要求             |
| ------------ | -------------------- |
| 查看项目列表 | 项目成员             |
| 查看文档列表 | 项目成员             |
| 同步项目文档 | 项目管理员           |

```mermaid
        CollabServer -->|防抖自动保存| OSS[阿里云 OSS-冷数据/持久化]
        APIGateway -->|转发提示词| LLM[大模型 API -DeepSeek/OpenAI]
    end

    subgraph "前端编辑器 Milkdown"
        UI[界面层] -->|渲染| Plugin_MD[Markdown 核心]
        UI -->|Ghost Text| Plugin_AI[AI 补全插件]
        UI -->|光标/锁| Plugin_Collab[协同插件]
    end
```

### 5.2 技术选型

| 层级           | 技术              | 说明                                |
| -------------- | ----------------- | ----------------------------------- |
| **前端编辑器** | Milkdown + Vue    | 基于 ProseMirror 的 Markdown 编辑器 |
| **协同引擎**   | Y.js              | CRDT 算法实现无冲突合并             |
| **协同服务**   | Collab Runtime    | 平台级 WebSocket 协作运行时，内部默认 provider 为 Hocuspocus |
| **缓存层**     | Redis             | 热数据存储和协同状态缓存            |
| **持久化**     | 阿里云 OSS        | 文档冷数据存储                      |
| **AI 服务**    | DeepSeek / OpenAI | 文档智能辅助                        |
| **内容框架**   | Nuxt Content      | 发布文档渲染                        |

### 5.3 数据流设计

```mermaid
sequenceDiagram
    participant Client as 客户端
    participant WebSocket as Collab Runtime
    participant Redis as Redis
    participant OSS as 阿里云 OSS

    Note over Client,OSS: 文档打开流程
    Client->>WebSocket: 请求打开文档
    WebSocket->>Redis: 检查热数据
    alt Redis 有缓存
        Redis-->>WebSocket: 返回 Y.js 状态
    else Redis 无缓存
        WebSocket->>OSS: 下载 .md 文件
        OSS-->>WebSocket: 返回文档内容
        WebSocket->>WebSocket: 转换为 Y.js 状态
        WebSocket->>Redis: 缓存热数据
    end
    WebSocket-->>Client: 返回文档状态

    Note over Client,OSS: 文档保存流程
    Client->>WebSocket: 编辑操作
    WebSocket->>Redis: 更新协同状态
    Note over WebSocket: 防抖 5 秒
    WebSocket->>OSS: 导出 Markdown 并上传
```
