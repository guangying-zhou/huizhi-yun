# Align 产品定位

状态：Draft  
日期：2026-04-30

## 1. 定位一句话

`Align` 不再作为全平台统一工作入口。统一员工入口、应用中心、轻量待办、通知公告和简单事项入口由 `console.employee-portal` 承接。

`Align` 保留为未来可选的深度组织协同应用，用于承接 Console 轻量协同无法覆盖的完整协同业务对象。

## 2. 为什么不再由 Align 充当第一入口

SSO 与 Console OIDC 落地后，企业员工的默认入口应该放在 `console`：

- `console` 是客户侧基础运行服务，天然承接本地登录会话、目录、应用中心和基础工作台。
- `platform` 继续作为控制面，不面向普通员工日常作业。
- `foundation` 已提供 AppLauncher、UserMenu、OIDC 登录/登出和共享壳层。
- 业务应用保持独立部署，通过统一入口和 SSO 形成一体化体验。

如果继续让 `Align` 做第一入口，会造成两个问题：

- `console` 与 `align` 都在做员工工作台，入口职责重复。
- 轻量待办、通知、公告和应用切换被迫沉淀到业务应用里，削弱 Console 作为基础运行服务的边界。

因此，门户层职责收敛到 `console`，`Align` 只保留业务层职责。

## 3. Console 与 Align 的边界

### 3.1 Console 员工入口层

Console 负责员工每天打开系统时需要看到的轻量入口：

- 我的应用
- 我的待办
- 通知公告摘要
- 最近访问
- 常用 / 置顶应用
- 简单事项入口
- 个人资料与统一登出
- 企业目录入口

这些能力是入口、摘要、提醒和跳转，不构成复杂业务主数据。

### 3.2 Align 深度协同层

Align 只在需要完整业务对象时启用：

- 跨部门事项协助单
- 人员借调申请与履约
- 协同 SLA 与统计分析
- HR 轻流程
- 财务轻流程
- 会议与纪要追踪中的深度协同对象

这些能力需要独立状态机、台账、报表、审批回写和跨模块业务键，因此不应默认塞进 Console。

## 4. 启用条件

当前阶段不建议投入 Align MVP。满足以下条件时再启动：

- Console 轻量事项出现完整生命周期。
- 业务需要协助单、借调单、HR/财务流程台账。
- 协同对象需要 SLA、统计、报表或组织效率分析。
- 客户希望把组织协同作为可选增强应用购买或独立部署。

## 5. 设计原则

- `Console` 是员工入口，不是深度协同业务应用。
- `Align` 是深度协同业务应用，不是全平台门户。
- 轻量入口状态留在 Console，例如置顶、最近访问、通知阅读状态和待办摘要。
- 完整协同业务对象留在 Align，例如协助单、借调单和 HR/财务轻流程。
- `Align` 不复制 Aims、Altoc、Assets、Codocs 的主数据，只保存稳定业务键和协同过程状态。

## 6. 实施建议

1. 第一阶段只建设 Console 统一员工入口。
2. Console 内实现应用中心、待办摘要、通知公告、最近访问和轻量事项入口。
3. `Align` 模块文档和 manifest 保留，但不作为 Starter 第一阶段必选应用。
4. 当轻量协同超出 Console 边界时，再基于 `Align` 独立建设深度组织协同能力。
