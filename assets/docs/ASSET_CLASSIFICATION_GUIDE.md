# Assets 资产分类口径

状态：DRAFT  
日期：2026-04-18  
适用范围：`Assets` 模块资产分类、页面设计、数据建模、联调口径

## 1. 文档目的

本文件用于统一 `Assets` 的资产分类口径，回答三个问题：

- 五大类资产分别是什么
- 不同类型资产之间如何划分边界
- 遇到容易混淆的对象时，应如何落类

## 2. 总体判断

`Assets` 当前采用 `实物 / 资源 / 产品 / 知识产权 / 数字` 五大类资产，本方向是合理的，应继续保持稳定。

核心原则不是继续新增一级大类，而是坚持：

- `资产本体分类` 回答“这是什么资产”
- `横向管理视图` 回答“这些资产在什么场景下被组织和使用”

因此：

- `环境视图`
- `客户交付视图`
- `安全视图`

都不应提升为新的一级资产分类，而应作为横向关系视图存在。

## 3. 五大类资产定义

### 3.1 实物资产

**定义**：公司持有或管理的有形对象。

**关注重点**：

- 采购
- 入库
- 领用
- 保管
- 归还
- 盘点
- 报废

**典型对象**：

- 笔记本电脑
- 显示器
- 办公手机
- 本地服务器
- 交换机
- 路由器
- GPU 工作站

**不包含**：

- 云主机实例
- SaaS 席位
- API 配额
- 代码仓库
- 文档库

#### 3.1.1 实物资产子类（定稿）

维护入口：

- `系统管理 -> 资产类别管理`
- 五大类资产的一级分类统一存储在数据库中，由管理页通过 Tabs 分 scope 维护
- 其中只有实物资产当前启用二级细类，其它大类暂按一级分类管理
`Assets` 中的实物资产，建议采用以下 6 个子类：

1. `办公设备`
2. `办公家具与设施`
3. `IT基础设施资产`
4. `交通运输资产`
5. `其他专业资产`
6. `低值物资`

各子类说明如下。

##### 办公设备

**定义**：员工或部门日常办公直接使用的终端与通用电子设备。

**典型对象**：

- 笔记本
- 台式机
- 显示器
- 手机
- 平板
- 扫描仪
- 通用测试机

**常见二级细类**：

- 笔记本
- 台式机
- 显示器
- 手机
- 平板
- 扫描仪
- 通用测试机

##### 办公家具与设施

**定义**：办公场所中的家具类和通用配套设施类实物。

**典型对象**：

- 办公桌
- 办公椅
- 文件柜
- 会议桌
- 投影仪
- 打印机
- 空调
- 饮水设备

**常见二级细类**：

- 办公桌
- 办公椅
- 文件柜
- 会议桌
- 投影仪
- 打印机
- 空调
- 饮水设备

##### IT基础设施资产

**定义**：承载本地机房、私有化部署、网络接入与基础算力的实体基础设施。

**典型对象**：

- 服务器
- GPU 设备
- 机柜
- UPS
- NAS
- 交换机
- 路由器
- 防火墙

**常见二级细类**：

- 服务器
- GPU 设备
- 机柜
- UPS
- NAS
- 交换机
- 路由器
- 防火墙

##### 交通运输资产

**定义**：用于公务、交付、巡检、搬运等场景的交通和运输类实物。

**典型对象**：

- 公务车
- 交付服务车
- 运维巡检车
- 电动车
- 叉车
- 搬运车

**常见二级细类**：

- 公务车
- 交付服务车
- 运维巡检车
- 电动车
- 叉车
- 搬运车

##### 其他专业资产

**定义**：不属于办公、基础设施、运输范畴，但具有明确专业用途的实物。

**典型对象**：

- 行业专用设备
- 客户交付专用硬件
- IoT 终端
- 研发实验设备
- 样机

**常见二级细类**：

- 行业专用设备
- 客户交付专用硬件
- IoT 终端
- 研发实验设备
- 样机

##### 低值物资

**定义**：单价较低、消耗或周转频率高、需要库存化管理的物资对象。

**典型对象**：

- 鼠标
- 键盘
- 网线
- 转接器
- 小型工具
- 常用备件
- 常规办公消耗类物品

**常见二级细类**：

- 鼠标
- 键盘
- 网线
- 转接器
- 小型工具
- 常用备件
- 办公耗材

说明：

- `低值物资` 可以纳入 `Assets` 统一管理范围
- 但不应与前 5 类共用完全相同的生命周期模型
- 它更适合走“库存、批次、出入库、领料、补货、盘点”流程
- 不应默认要求“一物一卡”或按固定资产方式折旧管理

#### 3.1.2 实物资产两层分类规则

实物资产建议统一采用两层分类：

- 一级：`asset_items.asset_subtype`
- 二级：`asset_physical_details.physical_type`

含义分别为：

- 一级分类回答“这类实物归属于哪一类管理口径”
- 二级分类回答“它具体是什么对象”

示例：

- `办公设备 / 笔记本`
- `办公家具与设施 / 办公桌`
- `IT基础设施资产 / 服务器`
- `IT基础设施资产 / NAS`
- `交通运输资产 / 公务车`
- `低值物资 / 鼠标`

约束：

- 一级分类必须稳定，用于统计、权限、流程和报表
- 二级细类用于录入、筛选、搜索和精细化展示
- 不建议把服务器、NAS、交换机这类对象直接提升为实物一级分类

#### 3.1.3 设备编码规则

实物资产的 `asset_code` 同时作为设备编码使用，不再单独维护第二套编码字段。

建议规则：

- 格式：`DV-{一级分类编码}-{二级细类编码}-{日期}-{随机后缀}`

示例：

- `DV-OFF-LTP-20260418-A1B2`
- `DV-ITI-SRV-20260418-C3D4`
- `DV-FAC-DSK-20260418-E5F6`

含义：

- `DV`：设备类资产前缀
- 一级分类编码：如 `OFF / FAC / ITI / TRN / PRO / CON`
- 二级细类编码：如 `LTP / SRV / DSK / CAR / NAS`
- 日期：生成日期
- 随机后缀：保证唯一性

补充规则：

- 设备编码一经生成，不建议人工频繁修改
- 条形码编码值默认直接使用 `asset_code`
- 二维码内容默认使用 `HZY-ASSET:{asset_code}`

### 3.2 资源资产

**定义**：支撑研发、运行、交付、客户服务持续提供能力的生产性资源。

**关注重点**：

- 开通
- 激活
- 分配
- 配额
- 续费
- 到期
- 释放
- 成本

**典型对象**：

- 云服务器
- 数据库实例
- Kubernetes 集群
- 对象存储
- SaaS License
- 开发工具席位
- 短信包
- 模型 API Key
- Token 配额
- 域名
- SSL 证书
- VPN / WAF / 堡垒机

**不包含**：

- 本地采购回来的实体设备本体
- 产品主档
- 代码或文档本身

### 3.3 产品资产

**定义**：面向内部或外部交付的业务能力主档。

**关注重点**：

- 产品主档
- 产品分级
- 产品负责人
- 技术底座
- 关联资源
- 关联数字资产
- 关联知识产权

**典型对象**：

- Aims
- Altoc
- Assets
- Align
- 某客户私有化交付产品线
- 共用技术底座

**不包含**：

- Git 仓库
- API 文档
- 安装包
- 商标证书
- 云资源实例

说明：

- `产品` 是“业务能力”和“交付单元”
- `技术底座` 不是新的一级分类，应作为产品资产下的重要子域管理

### 3.4 知识产权资产

**定义**：公司依法拥有、申请或持续维护中的权利型资产。

**关注重点**：

- 权属
- 申请状态
- 有效期
- 续展
- 关联产品
- 关联品牌

**典型对象**：

- 软件著作权
- 商标
- 专利
- 资质证照

**不包含**：

- 产品主档本身
- 代码仓库
- 设计稿
- 合同

### 3.5 数字资产

**定义**：研发、交付、运维过程中形成的可复用数字成果物。

**关注重点**：

- 目录化登记
- 存储位置
- 所有人
- 访问范围
- 关联产品
- 关联环境
- 关联文档

**典型对象**：

- Git 仓库
- 需求文档
- API 文档
- 设计稿
- 数据集
- 提示词模板
- 模型文件
- 安装包
- 部署脚本
- 初始化数据

**不包含**：

- 云资源实例
- 席位或订阅
- 产品主档
- 权利凭证

## 4. 最容易混淆的三组边界

### 4.1 产品 vs 数字

判断规则：

- `产品` 回答“我们对内或对外交付什么能力”
- `数字` 回答“这个能力背后有哪些数字成果物”

因此：

- `Aims` 是产品资产
- `Aims` 的代码仓库是数字资产
- `Aims` 的需求文档是数字资产
- `Aims` 的安装包是数字资产

结论：产品不是仓库，不是文档，也不是安装包。

### 4.2 资源 vs 数字

判断规则：

- `资源` 回答“系统靠什么运行和交付”
- `数字` 回答“系统沉淀了什么内容和成果”

一个简单识别方法：

- 会涉及开通、续费、到期、席位、配额、费用的，多数是资源资产
- 会涉及存储位置、访问范围、版本归档、成果复用的，多数是数字资产

因此：

- 云服务器是资源资产
- GitLab 席位是资源资产
- 模型 API Key 是资源资产
- Git 仓库是数字资产
- 部署脚本是数字资产

### 4.3 实物 vs 资源

判断规则：

- `实物` 管“拥有关系、保管关系、盘点关系”
- `资源` 管“生产能力、服务能力、运行能力”

对于部分对象，允许双重建模，但语义必须不同。

例如：

- 一台本地 GPU 服务器
  - 作为服务器设备，属于实物资产
  - 作为算力提供节点，属于资源资产

要求：

- 不把两种语义混成一条记录
- 两条记录之间应建立关联关系

## 5. 横向视图与资产分类的关系

以下对象不是一级资产分类：

- 环境
- 客户交付
- 安全

它们属于横向管理视图。

### 5.1 环境视图

环境由以下对象组合形成：

- 资源资产
- 数字资产
- 产品版本
- 安全配置

环境回答的是“某个可运行单元由哪些东西构成”，不是“环境本身是什么资产分类”。

### 5.2 客户交付视图

客户交付视图用于按客户、合同、项目、环境组织资产关系。

它回答的是：

- 某次交付用了哪些资产
- 成本归给谁
- 运维责任在谁

它不是一个新的资产大类。

## 6. 分类判定规则

实际落类时，优先按以下顺序判断：

1. 它是不是有形对象  
   是：优先判为 `实物资产`
2. 它是不是持续提供运行/交付能力的资源  
   是：优先判为 `资源资产`
3. 它是不是对内对外交付的业务能力主档  
   是：判为 `产品资产`
4. 它是不是依法持有的权利凭证  
   是：判为 `知识产权资产`
5. 它是不是可复用的数字成果物  
   是：判为 `数字资产`

如果一个对象同时满足多项条件，按“本体语义优先、关系单独建模”的原则处理，不新增一级分类。

对于实物对象，还应继续判断其属于“资产型”还是“物资型”：

- 前 5 类实物子类默认按 `资产型` 管理
- `低值物资` 默认按 `物资型` 管理

判断规则：

- 需要按单件追踪责任人、位置、归还、寿命、维修、报废的，优先按 `资产型`
- 更关注库存数量、批次、仓库、出入库、补货的，优先按 `物资型`

## 8. 首版研发建议

分类体系保持五类不变，但实施节奏不应平均用力。

### 8.1 P0 做深

- 实物资产
- 资源资产
- 环境视图
- 客户交付视图

原因：

- 流程闭环最强
- 成本和预警价值最直接
- 与采购、分配、归还、到期管理强相关

### 8.2 P1 做准

- 产品资产
- 知识产权资产
- 数字资产

原因：

- 首版重点应放在“可登记、可查询、可关联”
- 不宜在 P0 过度追求复杂治理功能

## 9. 研发执行约束

为避免后续实现跑偏，建议遵守以下约束：

- 不新增第六类、第七类一级资产，除非业务语义已经明显超出五类框架
- 不把 `环境`、`客户交付`、`安全` 直接做成资产大类
- 不把 `技术底座` 升级成独立一级分类
- 不把 `产品`、`数字`、`资源` 混成同一类页面或同一张详情语义
- 对双重建模对象，明确“本体记录”和“关系记录”的语义差异
- `低值物资` 虽可纳入 `Assets`，但默认不走固定资产式生命周期
- 实现时应区分 `资产型` 与 `物资型` 两类流程，不要共用同一套状态机

## 10. 一句话结论

`Assets` 的五大资产分类应保持稳定，后续工作的重点不是继续加类，而是把 `产品 / 数字 / 资源` 的边界定清，并通过 `环境视图`、`客户交付视图` 承接跨类组合关系。
