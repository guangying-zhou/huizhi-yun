## 1. 文档目的

本文档用于明确企业资产管理系统的首版建设目标、范围边界、核心业务流程、数据模型、权限控制、功能优先级与验收标准，作为产品、设计、研发、测试和实施的共同依据。

本版在原有方案基础上，结合汇智云面向中小软件企业的一体化平台定位，对资产分类体系进行了重构：在保留主干分类的前提下，将 `云资源` 升级为 `资源资产`，并新增 `环境视图`、`客户交付视图`、`安全视图` 三类横向管理视图，以更好支撑研发、交付、运维、安全和经营协同。

## 2. 项目背景与目标

### 2.1 背景

随着公司业务规模扩大，企业资产种类日益增多，涵盖自研软件产品、实物设备、知识产权、数字资产、资源资产及其订阅授权等多个维度，目前面临以下问题：

- **台账缺失**：缺乏统一的资产登记与管理入口，资产状态不透明
- **归属不清**：资产负责人、使用人、保管人边界模糊，客户交付与公司自用经常混在一起
- **成本失控**：软硬件采购、资源订阅、License、API、AI 额度等费用缺乏系统性管控
- **预警不足**：资源、域名、证书、席位、Key、License 到期或额度不足无统一预警
- **追溯困难**：资产采购、分配、变更、回收、报废缺少完整记录
- **环境不可见**：客户环境、测试环境、生产环境往往依靠 Excel 或人工记忆维护，资源与环境关系不清
- **交付链路断裂**：客户、合同、项目、环境、资产之间缺乏统一关联，难以支撑交付、验收和运维

### 2.2 建设目标

1. **建立统一资产台账**：覆盖产品资产、实物资产、知识产权资产、数字资产、资源资产，并重点支持订阅、席位、API 配额、AI 资源、域名证书、安全资源等软件企业高频对象
2. **建立责任链路**：明确资产负责人、使用人、保管人、归属部门、归属项目、关联合同与服务环境
3. **打通关键流程**：实现采购申请、审批、入库/激活、分配、领用、归还、释放、报废/停用闭环
4. **建立环境与交付视角**：围绕环境视图和客户交付视图，串联客户、合同、项目、环境、资产关系
5. **建立预警机制**：对到期、余量不足、超期未归还、席位超分配、离职未回收等风险进行自动提醒
6. **支撑成本归因**：区分公司自用与客户交付，支持项目、部门、产品线、客户环境维度统计
7. **支撑平台一体化**：为后续与 Account、Aims、Codocs、Altoc、Align 的组织、项目、合同、审批、知识文档打通奠定统一资产模型基础

### 2.3 成功指标

上线后 3 个月内，建议以以下指标作为首版验收标准：

| 指标                         | 目标值 |
| ---------------------------- | ------ |
| 实物资产建账覆盖率           | >= 95% |
| 资源资产及订阅授权建账覆盖率 | >= 90% |
| 环境关联覆盖率               | >= 80% |
| 到期预警触达率               | >= 95% |
| 资产操作留痕覆盖率           | 100%   |
| 项目/部门成本归因覆盖率      | >= 80% |

## 3. 产品范围

### 3.1 V1 建设范围

#### P0 必做

- 实物资产台账
- 资源资产台账（含 SaaS License、席位、API 额度、AI 资源、域名证书、安全接入资源）
- 环境视图
- 客户交付视图
- 采购申请、审批、入库/激活
- 资产分配、领用、归还、释放、报废/停用
- 到期预警、配额告警、超期未归还提醒、席位与回收提醒
- 基础报表与操作日志
- Excel 批量导入导出

#### P1 次优先

- 产品资产台账
- 知识产权资产台账
- 数字资产目录化登记
- 财务科目映射
- 安全视图
- 移动端扫码盘点/领用确认
- 入转调离资产回收任务联动

#### P2 后续扩展

- 对接 OA/ERP 审批与财务科目
- 对接云厂商账单 API 自动回填费用
- 对接 GitLab API 自动同步代码仓库信息
- 对接 SaaS 平台席位使用数据
- 自定义资产模板与字段配置中心
- 更细颗粒度的 BI 分析与仪表盘

### 3.2 非范围

以下能力不纳入首版交付：

- 完整财务记账与折旧系统
- 资源自动开通、自动续费、自动释放
- 代码仓库、文档库、数据集的深度治理
- 合同管理系统替代
- 固定资产盘点硬件终端建设
- 全自动安全治理与自动处置体系

## 4. 术语定义

| 术语         | 说明                                                                                                         |
| ------------ | ------------------------------------------------------------------------------------------------------------ |
| 资产         | 公司持有、租用或管理的可识别对象，包括可归档的本体资产和可管理的关系对象                                     |
| 自用资产     | 由公司购买或租用，服务于内部经营活动                                                                         |
| 客户交付资产 | 为客户项目采购或开通、最终交付客户使用或为客户提供服务的资产                                                 |
| 资源资产     | 支撑研发、运行、交付、客户服务持续提供能力的生产性资源集合，既包括云上资源，也包括本地、私有化与混合部署资源 |
| 环境         | 由一组资源资产、数字资产、产品版本、安全配置构成的可运行单元，如测试环境、预发环境、客户生产环境             |
| 环境视图     | 按环境维度组织资产关系的管理视图，用于查看某环境由哪些资源和文档构成                                         |
| 客户交付视图 | 按客户、合同、项目、环境组织资产的管理视图，用于支撑交付、验收、运维与成本归因                               |
| 安全视图     | 按敏感级别、对外暴露程度、关键程度聚合资产的管理视图                                                         |
| 负责人       | 对资产完整性、有效性、续期负责的人                                                                           |
| 使用人       | 实际使用该资产的人                                                                                           |
| 保管人       | 对实物资产日常保管负责的人                                                                                   |
| 成本归因     | 将资产成本归属到部门、项目、产品线、客户环境等维度                                                           |

## 6. 资产分类体系

### 6.1 分类设计原则

本系统建议采用 `资产本体分类 + 横向管理视图` 的双层设计：

- `资产本体分类` 回答“这是什么资产”
- `横向管理视图` 回答“该资产服务谁、部署在哪、交付到哪、风险多高”

设计原则如下：

- 一级分类保持稳定，不因局部业务场景频繁新增大类
- 复杂对象优先通过视图、关系、标签表达，而非强行升格为一级资产
- 分类以业务管理视角为准，不直接等同于会计科目
- 会计科目、成本中心等通过映射字段关联
- 首版实施时，P0 分类必须完整闭环，P1 分类先满足“可登记、可查询、可关联”
- 优先服务中小软件企业的主链路：`产品 -> 项目 -> 环境 -> 资源 -> 交付 -> 运维`

### 6.2 资产本体分类总览

```
企业资产
├── P1 产品资产
├── P0 实物资产
├── P1 知识产权资产
├── P1 数字资产
└── P0 资源资产
```

一级分类说明：

- `产品资产`：承载业务价值的软件产品、版本、技术底座
- `实物资产`：公司持有或控制的有形设备
- `知识产权资产`：软著、商标、专利、版权、资质证照等权利型资产
- `数字资产`：代码、文档、数据、设计、模型、交付物等可复用成果
- `资源资产`：承载研发、运行、交付和客户服务能力的生产性资源

### 6.3 横向管理视图

以下对象建议作为横向视图管理，而非新增一级资产：

```
横向管理视图
├── P0 环境视图
├── P0 客户交付视图
└── P1 安全视图
```

说明：

- `环境视图` 用于承载“测试环境、预发环境、客户生产环境”等运行单元
- `客户交付视图` 用于串联 `客户 -> 合同 -> 项目 -> 环境 -> 资产`
- `安全视图` 用于聚合高敏感、高风险、对外暴露或关键业务资产
- `环境资产`、`客户交付资产`、`安全资产` 不建议单独设为一级分类

### 6.4 统一资产主模型

所有资产类型建议共享以下主字段：

### 6.5 产品资产

**定位**：公司对外交付或销售的自研软件产品、平台、SaaS 服务，以及支撑产品复用与演进的技术底座。

产品资产建议拆分为两层：

- `P1 产品主档`：面向公司级产品家底梳理、分类分级、责任归属
- `P2 产品版本与产品交付`：面向可发布版本、软件物料清单、客户交付关系与实施状态

同时建议增加一类配套栏目：

- `P1 技术底座`：面向基础平台、共用模块、中台能力、工具类模块的统一登记，用于构建产品依赖关系

#### 6.5.1 产品边界与管理规则

以下对象可纳入“产品资产”管理范围：

1. 能独立对外提供业务价值的产品或平台
2. 具有独立用户群体、业务目标或业务边界
3. 具备相对独立的代码仓库、数据库或部署实例
4. 有明确的业务负责人和技术负责人

建议补充以下管理规则：

- 命名规范：统一采用 `产品名-项目名` 或 `产品名-版本标识`
- 一项产品可对应多个产品版本，一个产品版本可对应多个产品交付
- 多个项目若共享同一产品能力，应挂接在同一产品主档下管理
- 依赖基础平台、中台能力或公共组件时，应记录依赖关系，不应重复建产品主档
- 基础平台、公共组件、工具模块优先登记到“技术底座”，再由产品或项目关联引用

- `研发项目` 用于关联内部研发立项、研发计划或迭代项目
- `实施项目` 属于产品交付层，用于关联面向客户的实施或交付项目，两者需区分

#### 6.5.5 技术底座

用于统一管理公司的平台项目、共用模块、工具类模块、中台能力，作为产品资产的依赖来源。

#### 6.5.6 产品治理视图

除台账字段外，建议在产品资产模块提供以下治理视图：

- `P1 产品地图视图`：按产品线、业务域查看产品分布
- `P1 产品依赖关系视图`：查看产品与技术底座、仓库、资源资产、知识产权的关联关系
- `P2 产品版本视图`：查看同一产品下的历史版本、模块组成与 SBOM
- `P2 产品交付视图`：查看版本与客户、合同、交付状态之间的关系
- `P2 技术治理视图`：查看文档完备度、仓库规范性、技术债、底座复用情况与重构计划

### 6.6 实物资产

**定位**：公司购置、持有或纳管的有形对象，包括办公设备、办公家具、机房与网络基础设施、车辆以及专业硬件等。

建议子类：

- `办公设备`
- `办公家具与设施`
- `IT基础设施资产`
- `交通运输资产`
- `其他专业资产`
- `低值物资`

建议采用两层分类：

- 一级分类：用于管理口径与统计归类
- 二级细类：用于具体对象录入与检索

示例：

- `办公设备 / 笔记本`

- `低值物资 / 鼠标`

| 资产类型     | 办公设备 / 办公家具与设施 / IT基础设施资产 / 交通运输资产 / 其他专业资产 / 低值物资 | P0     |

- 本地服务器、GPU 节点等对象，既可作为 `实物资产` 管理其所有权与保管关系，也可在 `资源资产` 中管理其提供的生产能力
- 同一现实对象允许在不同分类中建立关联记录，但两者语义不同，不应混为一张记录

### 6.7 资源资产

**定位**：支撑研发、运行、交付、客户服务持续提供能力的生产性资源集合。既包括云上资源，也包括本地、私有化或混合部署资源。

建议子类：

- `基础设施资源`：云主机、本地服务器算力、GPU 节点、存储、网络资源池
- `平台资源`：数据库、中间件、K8s 集群、CI/CD、监控、日志、备份
- `订阅与席位资源`：GitLab、企业邮箱、Figma、飞书增购、AI 工具席位
- `配额与接口资源`：短信包、OCR、地图 API、推送额度、第三方接口包
- `AI 资源`：模型 API Key、Token 包、向量库、推理实例、AI SaaS 席位
- `接入与安全资源`：域名、SSL 证书、DNS、WAF、VPN、堡垒机、漏洞扫描

| --------------------------------------- | -------------------------------------------- | ------------------------ |
| 环境资产                                | 归入环境视图                                 | 不单列一级资产           |
| 客户交付资产                            | 归入客户交付视图 + 归属类型=客户交付         | 本质是归属关系和交付关系 |
| 域名 / SSL 证书 / DNS                   | 资源资产 -> 接入与安全资源                   | 更接近运行接入资源       |
| WAF / VPN / 堡垒机 / 漏扫               | 资源资产 -> 接入与安全资源，同时进入安全视图 | 跨运维与安全管理         |
| 本地服务器 / GPU 节点                   | 实物资产 + 资源资产关联                      | 前者管设备，后者管产能   |
| GitLab / Figma / 邮箱 / AI 工具席位     | 资源资产 -> 订阅与席位资源                   | 重点管席位分配和回收     |

| 模型 API Key / 推理额度 / 向量库        | 资源资产 -> AI 资源                          | 重点管成本和安全         |
| 代码仓库 / 镜像仓库 / 安装包 / 部署脚本 | 数字资产                                     | 属于可复用数字成果       |
| 产品交付                                | 产品资产中的关系对象 + 客户交付视图          | 不建议设为一级资产       |

## 7. 核心业务流程

### 7.1 采购到入账主流程

```
采购申请 -> 审批 -> 下单 -> 到货/激活/开通 -> 入库/建账 -> 绑定项目/环境/客户 -> 分配/领用 -> 变更 -> 归还/释放 -> 报废/停用
```

### 7.2 实物资产流程

1. 发起采购申请
2. 审批通过后生成采购单
3. 到货后由资产管理员执行入库，生成资产编号
4. 分配给使用人并记录领用时间
5. 如发生转移、维修、归还，记录变更单
6. 到生命周期末端时执行报废，台账状态关闭

### 7.3 资源资产流程

1. 发起采购、开通申请或自建登记申请
2. 审批通过后记录订单信息或建设来源
3. 激活或登记资源并录入实例标识、到期时间、费用信息
4. 分配给部门、项目、个人、系统或环境
5. 如属于客户交付场景，需绑定客户、合同、项目和环境
6. 系统按规则发送到期、余量、超分配与回收预警
7. 续费、转移、释放、停用、密钥轮换需形成变更记录

### 7.4 环境与客户交付流程

1. 新建环境视图，选择环境类型（研发、测试、预发、内部生产、客户测试、客户生产）
2. 关联项目、产品版本、客户、合同等上下文信息
3. 将资源资产、数字资产、安全配置、关键文档挂接到环境
4. 如为客户交付场景，形成 `客户 -> 合同 -> 项目 -> 环境 -> 资产` 的交付链路
5. 在环境维度查看运行资源、交付文档、费用、到期信息和维护责任人

### 7.5 入转调离联动流程建议

1. 员工入职后，按岗位分配办公终端、基础席位、研发或业务工具权限
2. 员工转岗时，触发资产与席位的增减、转移和重新归属
3. 员工离职时，系统生成待回收任务，检查设备、账号、Seat、API Key、客户环境权限是否已回收
4. 离职回收完成后，资产状态更新为在库、待分配或已释放

说明：

- V1 可先支持手工创建回收任务与回收清单
- 与 Account、Align 的自动联动可在 P1/P2 逐步接入

### 7.6 状态机建议(通过Workflow实现)

#### 采购单状态

`待提交 -> 待审批 -> 已批准 -> 已下单 -> 已到货/已激活 -> 已入库 -> 已完成 -> 已关闭`

#### 实物资产状态

`待入库 -> 在库 -> 在用 -> 闲置 -> 维修中 -> 已报废`

#### 资源资产状态

`待开通 -> 生效中 -> 运行中 -> 即将到期 -> 已到期 -> 已续费 -> 已释放/已停用`

#### 预警状态

`待处理 -> 已确认 -> 已延期 -> 已完成 -> 已忽略`

## 8. 采购管理

### 8.1 供应商管理

涉及采购流程，建议建立供应商台账，避免采购单中的供应商名称自由录入导致数据不一致、资质不可追溯、历史采购价格无法复用。

#### 供应商建设原则

- `P0` 建立供应商基础台账，满足采购归档、关联、查询与统计
- `P1` 建立合格供应商台账，满足准入审核、资质管理和风险控制
- `P2` 再扩展供应商绩效、黑名单、年度复审等能力

#### 供应商基础台账字段

| 字段             | 说明                                                 | 优先级 |
| ---------------- | ---------------------------------------------------- | ------ |
| 供应商名称       | 统一名称，避免重复录入                               | P0     |
| 供应商编号       | 系统唯一编号                                         | P0     |
| 统一社会信用代码 | 企业唯一标识                                         | P0     |
| 供应商类型       | 硬件 / 软件 / 云服务 / AI 服务 / 安全服务 / 综合服务 | P0     |
| 联系人           | -                                                    | P0     |
| 联系方式         | 电话、邮箱等                                         | P0     |

### 8.3 入库与激活

- **P0 实物入库**：登记资产编号、SN、位置、保管人，自动创建设备台账

- **P1 客户交付直达项目**：采购完成后直接归属到客户、合同、项目、环境视图

## 10. 成本归因与报表

### 10.1 成本口径

首版建议采用以下口径：

1. 采购金额优先取实际采购单金额，无实际金额时取预算金额
2. 资源资产成本优先按月均费用统计，后续支持对接账单 API 替换
3. 共享资源先允许手工指定归属维度，后续再扩展自动分摊
4. 客户交付资产必须绑定项目，必要时绑定合同和环境
5. 本地/私有化资源如难以获取实时账单，V1 允许按月人工录入折算成本
